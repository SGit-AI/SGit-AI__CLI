## 1. Executive Summary

The codebase has completed the majority of the v0.16.3 architecture brief. Self-describing file IDs, sub-tree model, encrypted metadata, batch read API, and legacy code deletion are all in place. The remaining gaps are well-defined and mechanically achievable.

**Conformance: ~80% of the architecture brief is implemented.**

The 20% gap consists of five items, all identified in the debrief. This review validates the debrief's assessment, adds architectural detail, identifies two additional issues not in the debrief, and confirms the priority order.

## 3. Implementation Gaps

### Gap 1: Clone Uses `list_files()` Instead of Graph-Walking

**Spec says:** Clone follows the deterministic object graph: derive ref → read HEAD → walk commit → tree → sub-trees → blobs → read branch index → fetch keys → register.

**Code does:** `Vault__Sync.clone()` line 783: `self.api.list_files(vault_id, 'bare/')` followed by downloading every file individually. This is the pre-v0.16.3 approach.

**Impact:**
- Requires the `list` endpoint (which the spec says should eventually be authenticated)
- Cannot work with storage backends that don't support listing
- Downloads ALL objects including historical commits, not just HEAD-reachable ones
- No batch read optimization
- Doesn't match the simulation's clone request budget (4 + depth vs N requests)

**Priority:** HIGH — this is the largest remaining algorithmic change.

**Fix:** Rewrite `clone()` to:
1. Derive `ref_file_id` and `branch_index_file_id` from vault key (already available in `keys`)
2. Read HEAD ref → commit_id
3. Walk commit → tree → sub-trees → blobs (using `Vault__Sub_Tree` or new helper)
4. Read branch index at deterministic ID
5. Fetch keys referenced by branch index
6. Register new clone branch

### Gap 2: Push Does Not Create Named Branch Merge Commit

**Spec says (v0.13.30 section 6.3 step 6):** After pushing clone changes to the named branch, create a merge commit ON the named branch, signed with the named branch key. The named branch should have its own commit chain showing merges from clone branches.

**Code does:** `push()` updates the named ref to point directly at the clone commit (line 70-76 in `Vault__Batch`). The named branch ref is fast-forwarded — no merge commit, no provenance boundary.

**Impact:**
- Named branch history is indistinguishable from clone branch history
- No merge provenance — cannot tell which clone branch contributed which changes
- Named branch has no independent signature chain

**Note:** The pull/merge flow DOES create merge commits on the clone branch (line 386-392 in `Vault__Sync`). The gap is specifically on the push side — the named branch needs its own merge commit.

**Priority:** MEDIUM — self-contained ~30 lines. Should be done before graph-walking clone since it affects what clones will see.

**Fix:** In `push()`, after computing the delta and before the batch:
1. Load named branch private key from vault
2. Create merge commit: `parents=[named_commit_id, clone_commit_id]`, `tree_id=clone_tree_id`, signed with named key
3. Update the batch ref operation to point to the merge commit, not the clone commit

### Gap 3: `_upload_bare_to_server()` Uses `os.walk()` Instead of Explicit File Enumeration

**Spec says:** Upload specific objects, refs, indexes, and keys by enumerating known files.

**Code does:** `_upload_bare_to_server()` (line 1099-1131) walks the entire `bare/` directory with `os.walk()`.

**Impact:**
- Works correctly today, but is fragile — picks up any file in `bare/`, not just known vault objects
- Doesn't validate that uploaded files are actual vault objects
- Not aligned with the "server mirrors bare/ exactly" principle since it's driven by filesystem state, not object graph

**Priority:** LOW — functionally correct, architectural smell. Can be addressed alongside graph-walking clone.

### Gap 4: `_is_first_push()` Uses `list_files()`

**Spec says:** The `list` endpoint should eventually be authenticated or removed.

**Code does:** `_is_first_push()` (line 1091-1097) calls `self.api.list_files(vault_id, 'bare/')` to check if the vault has files on the server.

**Priority:** LOW — can be replaced with a health check endpoint (`GET /api/vault/health/{vault_id}`) or by trying to read the deterministic HEAD ref and interpreting 404 as "first push."

### Gap 5: No `local/tracking.json` for Fast LCA

**Spec says (v0.13.30 section 8.4, v0.16.3 addendum section 9):** Write `local/tracking.json` after every pull/push to record the last-known commit ID for each ref. Used as merge base for the fast path (avoids full commit chain walk for LCA).

**Code does:** No tracking file. LCA is computed by walking the full commit chain every time.

**Priority:** LOW for correctness (LCA works without it), MEDIUM for performance at scale.

## 4. Additional Issues Found (Not in Debrief)

### Issue A: `_checkout_tree()` Retains Flat-Tree Fallback (Dead Code)

**File:** `Vault__Sync.py` lines 863-891

`_checkout_tree()` has two code paths: sub-tree checkout (via `tree_id`) and flat-tree checkout (via `tree.entries`). The flat-tree path accesses `entry.path` and `entry.name` — fields that NO LONGER EXIST on `Schema__Object_Tree_Entry`. This is dead code that will crash if ever reached.

**Fix:** Delete the flat-tree fallback (lines 877-891). Only the `tree_id` path is valid.

### Issue B: `_entry_path()` References Removed Fields

**File:** `Vault__Sync.py` line 961-962

```python
def _entry_path(self, entry: Schema__Object_Tree_Entry) -> str:
    return str(entry.path) if entry.path else str(entry.name)
```

`entry.path` and `entry.name` no longer exist on the schema. This method will crash if called. It's still referenced in the dead flat-tree fallback code in `_checkout_tree()`.

**Fix:** Delete `_entry_path()` alongside the flat-tree fallback.

### Issue C: `Vault__Sync._remove_deleted_files()` Has Dead Flat-Tree Path

**File:** `Vault__Sync.py` lines 893+

The `_remove_deleted_files()` method has a branch for flat trees (`old_tree` / `new_tree` parameters) that accesses `self._entry_path(e)` — which will crash due to Issue B.

**Fix:** Remove the flat-tree branch. Only the `old_tree_id` / `new_tree_id` path is valid.

### Issue D: `Vault__Id` Regex — `[0-9a-f]` vs `[a-z0-9]`

**File:** `Safe_Str__Vault_Id.py` — regex `^[0-9a-f]{8}$`

The vault browser brief notes that vault IDs should match `[a-z0-9]{8}` (alphanumeric), not `[0-9a-f]{8}` (hex only). The current CLI regex is hex-only, which would reject vault IDs containing characters `g-z`.

**Impact:** If the server generates vault IDs with non-hex characters (e.g. `12cpxeq9` from the simulation), the CLI's `Safe_Str__Vault_Id` would reject them.

**Verification needed:** Check if the live vault ID `12cpxeq9` contains `p` and `q` — yes it does. The current regex `[0-9a-f]` would REJECT this valid vault ID.

**Fix:** Change regex to `^[a-z0-9]{8}$` to match the actual server behavior.

**Priority:** HIGH — blocks real vault operations.

## 8. Conclusion

The codebase is **architecturally sound** and well-aligned with the v0.16.3 brief. The self-describing ID convention, sub-tree model, encrypted metadata, and legacy cleanup are all correctly implemented. The remaining work is concentrated in two areas:

1. **Clone algorithm** (graph-walking instead of list-all) — the largest gap
2. **Push provenance** (named branch merge commit) — small but important for the signature chain

The Vault ID regex issue (Issue D) is a blocking bug for real-world use and should be fixed immediately.

All dead code paths (Issues A-C) should be cleaned up to prevent confusion and potential crashes.
