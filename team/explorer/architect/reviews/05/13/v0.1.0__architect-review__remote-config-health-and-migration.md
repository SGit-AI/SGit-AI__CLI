## 1. Executive Summary

The CLI today has a remote-handling surface that is **incomplete by design** and the failure mode Dinis hit is the worst-case manifestation of that incompleteness:

1. There **is** a `sgit vault remote add | remove | list` command tree, but `show` and `set-url` are missing, no health check exists, tokens are saved per-vault to a different store than the named remote, and the named-remote `vault_id` is decoupled from the active vault's identity (allowing rot).
2. There is **no first-class diagnostic command**. The `_prompt_remote_setup()` flow calls `list_files()` once to "verify" the token; when that fails (e.g. `Connection refused` from inside Apple Container), the CLI prints a warning, saves the token anyway, then proceeds into push which cannot succeed. The user sees a generic Python traceback ~3 lines later.
3. The `Workflow__Vault_Move` exists and **re-keys** the vault on move (good: forces re-encryption; bad: it is not a pure transport-layer migration). There is no equivalent "stream ciphertext as-is between servers" workflow, and there is no recovery path when the destination push partially completes.

This review proposes one coherent design that covers all three concerns without weakening the zero-knowledge guarantee. None of the proposed additions require new server endpoints **except one optional `/api/info` health probe** (used only by `sgit doctor`, not the protocol). All proposals respect:

- No tokens persisted in plaintext (use the existing encrypted `Secrets__Store`).
- No vault identifiers, names, or branches leaked through new endpoint surfaces.
- Type_Safe everywhere; no `dict`/`str` field stores; classes-for-everything for the doctor checks.

**Severity verdict:**
- The "warn-but-proceed" bug is **HIGH** (user-facing data-loss vector — the user thinks they pushed when they did not).
- Lack of `sgit doctor` is **MEDIUM** (workaround exists, but every self-hosted user hits the same cliff).
- Lack of stream-mode migration is **MEDIUM** (the re-key move covers the use case at higher cost).

## 3. Findings

### Finding 1 — `_prompt_remote_setup` swallows a fatal error as a warning (HIGH)

**Location:** `sgit_ai/cli/CLI__Vault.py:842-894`

**Observation:** The interactive setup at line 880-888 wraps the verification API call in a bare `except Exception as e:`, prints `Warning: could not verify token ({e})`, and then **proceeds to save the token and base_url** at lines 890-892. The push then fails with a `URLError` in a deeper call stack, surfaced by `_print_friendly_error()` as `error: network or I/O failure — <urlopen error [Errno 111] Connection refused>`. This is the exact symptom Dinis hit.

**Why it matters:** `Connection refused` against a TCP socket is **not** a transient or recoverable condition. It is proof that no listener exists at that address. The current code treats it as soft. From the user's perspective, the `Warning:` message implies "I'm continuing because the failure was non-fatal" — which is false; the failure is total.

**Recommendation:** Replace the catch-and-warn with **fail-fast classification**. The classification is:

| Exception kind | Class | Severity | UX action |
|---|---|---|---|
| `socket.gaierror` (DNS) | unreachable | FATAL | suggest checking the URL spelling; if host is loopback, run doctor |
| `ConnectionRefusedError` | unreachable | FATAL | suggest doctor; if host ∈ {0.0.0.0, 127.0.0.1, localhost}, point at the container-vs-host loopback gotcha |
| `socket.timeout` | unreachable | FATAL | suggest doctor (timeout=5s) |
| `ssl.SSLCertVerificationError` | unreachable | FATAL | re-route to existing `_check_ssl_error()` helper |
| `HTTPError(401|403)` | auth | FATAL | "token rejected — run sgit auth" |
| `HTTPError(404)` on `/api/info` | wrong-server | FATAL | "endpoint not found — is this an SG/API server?" |
| `HTTPError(5xx)` after retries | transient | RETRY/FATAL | accept warning if user passes `--ignore-health`; otherwise fatal |

The token must NOT be persisted unless verification returns success. No silent fallthrough.

**Severity:** HIGH (correctness; user-visible).

### Finding 2 — Three split stores for what is conceptually one record (MEDIUM)

**Locations:**
- `.sg_vault/local/remotes.json` (named multi-remote list — `Vault__Remote_Manager`)
- `.sg_vault/local/base_url` (single-line plaintext — `CLI__Token_Store`)
- `.sg_vault/local/token` (single-line plaintext — `CLI__Token_Store`)

**Observation:** A "remote" today is fragmented across three files. `remotes.json` knows `name`, `url`, `vault_id` but **not** the token; `local/token` knows the token but not which remote it belongs to. Effect: a vault can have an `origin` in `remotes.json` and a different token+url cached in `local/`. The push path reads from `local/`; the `cmd_remote_list` path reads from `remotes.json`. They drift.

**Why it matters:** When `sgit doctor` and `sgit vault migrate` need to operate on "the remote", they need a single source of truth. Two stores means two of every doctor check, two failure modes, and one of them silently wins.

**Recommendation:** Unify on a single `Schema__Remote_Config` record per remote name, stored in **one** of two locations:

1. **Per-vault plaintext (status quo)** for non-secret fields (`name`, `url`, `vault_id`, `is_default`, `tls_verify`, `last_health_at`, `last_health_status`).
2. **Encrypted host-wide secret store** for the access token, keyed by `(remote_url, vault_id)` and decrypted only on use.

The token must never live in plaintext on disk. Today's `chmod 0600` is not sufficient for a shared-host scenario (root can read; backup tools snapshot it; tokens leak through `cp -a`).

**Severity:** MEDIUM (latent — bites on doctor/migrate, not on today's push).

### Finding 3 — No `sgit doctor` exists (MEDIUM, but blocks Apple-Container UX)

**Location:** `sgit_ai/cli/CLI__Main.py` (no parser registered).

**Observation:** The only quasi-diagnostic capability today is `cmd_info` (`CLI__Vault.py:1149`), which prints vault identity and the resolved remote URL but does **no network probing**. The verification inside `_prompt_remote_setup` is a single `list_files()` call with no diagnostic differentiation between "DNS failed" / "no listener" / "TLS bad" / "wrong server" / "token rejected".

**Recommendation:** Add `sgit doctor` (or `sgit remote health <name>` as a sub-form) with a fixed ordered checklist. See §5 for the contract.

**Severity:** MEDIUM (every self-hosted user hits the same cliff; this is the right UX guard rail).

---

### Finding 4 — Container-loopback detection has no signal today (MEDIUM)

**Observation:** When `base_url` resolves to host ∈ {`localhost`, `127.0.0.1`, `0.0.0.0`, `::1`}, and the CLI is running inside a container, the loopback resolves to the container's loopback — not the host's. The CLI emits **no signal at all** about this. The `Errno 111 Connection refused` in the transcript is identical to "the host has no listener on 8080" in symptom, but the fix is different ("rebuild the URL with the container-host bridge").

**Recommendation:** In the doctor checklist, when host is loopback, detect container environment from filesystem markers and emit a specific suggestion. The detection signals:

| Marker | Indicates | Suggested replacement |
|---|---|---|
| `/.dockerenv` exists | Docker / Docker-Desktop | `host.docker.internal` |
| `/run/.containerenv` exists | Podman / Apple Container | `host.containers.internal` |
| `/proc/1/cgroup` contains `docker\|kubepods\|containerd` | Container runtime | host LAN IP |
| `$container=oci` env var | Generic OCI (incl. Apple Container) | host LAN IP |
| None of the above + loopback fails | Truly local but no listener | "is the server actually running on :PORT?" |

This is presentational ONLY; it never weakens the zero-knowledge boundary (nothing about the vault leaves the CLI on this probe).

**Severity:** MEDIUM (specific to the use case Dinis hit; will recur for every container user).

### Finding 5 — No transport-layer `sgit vault migrate` (MEDIUM)

**Observation:** `Workflow__Vault_Move` re-keys the vault. The 8 steps include `Derive_New_Keys`, `Build_Temp_Vault` (re-encrypts every object), and `Write_Sentinel_Commits`. This is a **safety feature** when the user suspects the old server may have copied ciphertext; it is **massive overhead** when the user is simply moving an encrypted vault from one trusted SG/API instance to another.

**Why it matters:**
- For a vault with 10k objects, re-key = decrypt + re-encrypt + new object IDs for everything; minutes to hours.
- A simple "stream the bytes" migration is O(N) read-write, no crypto, can re-use existing batch endpoints.
- Re-key changes the vault_id; every clone on every device has to re-clone. Stream-migration preserves the vault_id and old clones can switch by `sgit vault remote set-url`.

**Recommendation:** Keep `Workflow__Vault_Move` for the re-key case (rename it to `vault rekey-move`). Add a new `Workflow__Vault_Migrate` whose contract is in §6 of this review.

**Severity:** MEDIUM.

---

### Finding 6 — `Safe_Str__Base_URL` accepts ambiguous loopback strings (LOW)

**Location:** `sgit_ai/safe_types/Safe_Str__Base_URL.py`

**Observation:** The regex `[^a-zA-Z0-9:/._\-]` allows `http://0.0.0.0:8080` but does not parse the URL or distinguish scheme/host/port. The doctor command will want structured fields. Today's Safe_Str is fine for storage but the doctor's network layer needs a parsed view.

**Recommendation:** Introduce four new safe types (full definitions in §4.1) — `Safe_Str__URL_Scheme`, `Safe_Str__Host`, `Safe_UInt__Port`, and a `Schema__Parsed_URL` Type_Safe that holds the breakdown. The Base_URL stays as the canonical storage format.

**Severity:** LOW (cosmetic; no blocker until doctor lands).

---

### Finding 7 — `cmd_remote_add` mandates a `remote_vault_id` argument (LOW)

**Location:** `sgit_ai/cli/CLI__Main.py:425-430`

**Observation:** Today `sgit vault remote add origin <url> <vault_id>` requires the vault_id positionally. In 95% of cases, the user wants to register the **current vault's** ID — and we already know it from `vault_key`. The git mental model is `git remote add origin <url>`, no ID.

**Recommendation:** Make `remote_vault_id` optional. If omitted and the current directory is a vault, default to the local vault_id. Keep it required for the cross-vault case (which is real but rare).

**Severity:** LOW (UX polish).

---

### Finding 8 — No `default` remote concept; push only consults `local/base_url` (LOW)

**Observation:** The remote list supports multiple entries but the push path never asks `Vault__Remote_Manager` — it goes straight to `CLI__Token_Store.resolve_base_url()`. Effectively `remotes.json` is decorative today.

**Recommendation:** Add `is_default : bool` to `Schema__Remote_Config` (see §4.2), enforce at most one default, and have all sync paths consult `Vault__Remote_Manager.get_default(directory)` first. The legacy `local/base_url` becomes a one-time migration source (read it, write it as `origin` with `is_default=true`, delete the legacy file).

**Severity:** LOW (architectural cleanup, but it has to happen before doctor/migrate to keep one source of truth).

## 4. Contract — Remote Config

### 4.1 New safe types

```python
# sgit_ai/safe_types/Safe_Str__URL_Scheme.py
class Safe_Str__URL_Scheme(Safe_Str):
    regex             = re.compile(r'[^a-z]')
    max_length        = 8
    allow_empty       = False
    to_lower_case     = True
    # Validates one of {'http', 'https'} at construction; raises otherwise.

# sgit_ai/safe_types/Safe_Str__Host.py
class Safe_Str__Host(Safe_Str):
    # Permits DNS hostnames, IPv4 dotted-quads, and the bare loopback strings
    # 'localhost' / '0.0.0.0' / '127.0.0.1' / '::1'. IPv6 brackets handled separately.
    regex             = re.compile(r'[^a-zA-Z0-9.\-:_\[\]]')
    max_length        = 253
    allow_empty       = False
    to_lower_case     = True

# sgit_ai/safe_types/Safe_UInt__Port.py
class Safe_UInt__Port(Safe_UInt):
    min_value         = 1
    max_value         = 65535

# sgit_ai/safe_types/Safe_Str__Remote_Name.py
class Safe_Str__Remote_Name(Safe_Str):
    regex             = re.compile(r'[^a-z0-9_\-]')
    max_length        = 64
    allow_empty       = False
    to_lower_case     = True
    # Default value contract: 'origin' is the conventional name; not enforced.
```

### 4.2 Updated `Schema__Remote_Config`

```python
# sgit_ai/schemas/Schema__Remote_Config.py — REPLACE existing
class Schema__Remote_Config(Type_Safe):
    name              : Safe_Str__Remote_Name = None
    url               : Safe_Str__Base_URL    = None              # canonical storage
    vault_id          : Safe_Str__Vault_Id    = None
    is_default        : Safe_Bool             = None              # at most one default per vault
    tls_verify        : Safe_Bool             = None              # default True; override for self-signed
    created_at        : Safe_Str__ISO_Timestamp = None
    last_health_at    : Safe_Str__ISO_Timestamp = None            # set by `sgit doctor`
    last_health_status: Safe_Str__Health_Status = None            # 'ok' | 'degraded' | 'unreachable' | 'unknown'
```

Note: `Safe_Bool` does not exist today; the architecture-deep-review (Finding 2.3) already flagged the lack of one. Pending that decision, use a `Safe_Str__Bool_Token` ('true'/'false') as an interim. **This is an open question for Dinis — see §8.**

### 4.3 Token storage (encrypted)

The plaintext `.sg_vault/local/token` file goes away. Replacement:

```python
# Tokens live in the existing host-wide encrypted store:
#   ~/.sg-send/vaults.enc  (Secrets__Store, AES-GCM, master key from passphrase)
#
# Token key format: 'remote-token:{remote_url}:{vault_id}'
# Stored value:     the raw access token string
#
# Reasoning:
#   - One token may apply to multiple vaults on the same server.
#   - One vault may need different tokens on different servers (after migrate).
#   - Keying on (url, vault_id) captures both axes.
```

The on-disk per-vault `.sg_vault/local/token` is migrated on first run after upgrade: read, write into the encrypted store, secure-unlink the old file.

If no passphrase is configured (Secrets__Store unset), tokens fall back to `~/.sg-send/tokens/{sha256(url:vault_id)[:16]}` with `chmod 0600` and a one-time `WARNING:` printed on save. This is strictly weaker than encrypted-at-rest; it is the "no-passphrase" escape hatch.

### 4.4 Command surface

| Command | Behaviour |
|---|---|
| `sgit vault remote add <name> <url> [<vault_id>] [--default] [--token <T>] [--no-verify-tls] [--no-health-check]` | Add. Runs `sgit doctor` against the new remote unless `--no-health-check`. Fails if not OK. `<vault_id>` defaults to current vault's. |
| `sgit vault remote list [--json]` | List all remotes with last health status. Marks default with `*`. |
| `sgit vault remote show <name>` | Print full record (token redacted to `***...(N chars)`); print full URL, vault_id, default flag, last health. |
| `sgit vault remote rm <name>` | Remove the named remote. Refuses if it is the default and others exist (force with `--force`). Token deleted from encrypted store. |
| `sgit vault remote set-url <name> <new-url>` | Rewrite URL. Runs doctor against new URL before accepting. Token unchanged. |
| `sgit vault remote set-default <name>` | Flip the default flag (clears prior default). |
| `sgit vault remote rename <old> <new>` | Rename. Token key updated. |

All paths route through a single `Vault__Remote_Manager` (no parallel `CLI__Token_Store` path).

### 4.5 Zero-knowledge audit

The server sees nothing new. The remote config is purely client-side. The encrypted token store decryption happens at process start; the decrypted token lives only in CLI memory and the HTTP header for the request lifetime.

**Verdict:** no new leak surface.

## 5. Contract — `sgit doctor`

### 5.1 Command surface

```
sgit doctor [--remote <name>] [--json] [--timeout <seconds>] [--no-color]
```

- No `--remote` flag: doctor runs against the **default remote** of the current vault.
- `--json` outputs a `Schema__Doctor_Report` instance as JSON. (Useful for CI.)
- `--timeout` defaults to 5 seconds per check.

### 5.2 Schema__Doctor__Check

```python
# sgit_ai/schemas/Schema__Doctor__Check.py
class Schema__Doctor__Check(Type_Safe):
    name        : Safe_Str__Step_Name      = None   # 'dns_resolve', 'tcp_reachable', ...
    status      : Enum__Doctor_Status      = None   # PASS | FAIL | WARN | SKIP
    duration_ms : Safe_UInt__Vault_Version = None
    message     : Safe_Str__Error_Message  = None
    hint        : Safe_Str__Error_Message  = None   # optional suggested fix
```

```python
# sgit_ai/schemas/Schema__Doctor__Report.py
class Schema__Doctor__Report(Type_Safe):
    remote_name : Safe_Str__Remote_Name    = None
    remote_url  : Safe_Str__Base_URL       = None
    started_at  : Safe_Str__ISO_Timestamp  = None
    overall     : Enum__Doctor_Status      = None
    checks      : list[Schema__Doctor__Check]
```

```python
# sgit_ai/safe_types/Enum__Doctor_Status.py
class Enum__Doctor_Status(Enum):
    PASS = 'pass'
    WARN = 'warn'
    FAIL = 'fail'
    SKIP = 'skip'
```

### 5.3 Check sequence (fixed order, fail-fast)

| # | Check name | What it does | Pass | Fail message |
|---|---|---|---|---|
| 1 | `parse_url` | `urllib.parse.urlparse(url)` → scheme, host, port | scheme ∈ {http, https}; host non-empty | "URL is malformed: ..." |
| 2 | `dns_resolve` | `socket.getaddrinfo(host, port)` | At least one address returned | "DNS resolution failed for `<host>`" (skip if host is literal IP) |
| 3 | `tcp_reachable` | `socket.create_connection((host, port), timeout=t)` | Connection accepted | "Cannot connect to `<host>:<port>` — connection refused / timeout" |
| 4 | `loopback_container_warn` | If host ∈ loopback AND container markers present (see §3 Finding 4) | always PASS (warning only emitted on prior FAIL of 3) | n/a |
| 5 | `tls_handshake` | If scheme=https: `ssl.create_default_context().wrap_socket(...)` | Handshake completes, cert validates | "TLS handshake failed: `<err>`" (route to `_check_ssl_error` for cert-chain hints) |
| 6 | `api_info` | `GET {url}/api/info` (NEW server endpoint, optional) | HTTP 200, JSON includes `service: 'sgraph-vault'` | "Endpoint did not return SG/API service info — is this the right URL?" |
| 7 | `token_verify` | `GET {url}/api/auth/whoami` (NEW server endpoint, optional) with the configured token | HTTP 200; response contains identity | "Token rejected (HTTP 401)" / "Token has no write scope" |
| 8 | `vault_known` | `GET {url}/api/vault/list/<vault_id>` (existing) — does the vault exist on this server? | HTTP 200, list returned (may be empty for first-push) | WARN: "Vault not found on remote — will be created on first push" |
| 9 | `write_probe` | OPTIONAL via `--write-probe`. Does a write+delete of a single ephemeral object `obj-cas-imm-{random}` under a probe-specific key. | Round trip OK | "Write probe failed: `<err>`" |

**Important:**
- Checks 6 and 7 require NEW server endpoints. Both are **read-only metadata**; neither carries vault payload. If the server is older and returns 404, the check emits WARN (not FAIL). The CLI degrades gracefully.
- Check 9 is opt-in because it writes encrypted bytes to the server. The probe payload is random AES-GCM ciphertext of the literal string `b'sgit-doctor-probe'` under a throwaway random key (not the vault's read_key). The probe file_id is `obj-cas-imm-<random12hex>` so it sits in the same object namespace as real objects but cannot collide with a real content hash (random vs. SHA-256 of content). The probe is deleted on success; on failure the orphan is left for human cleanup.
- Check 8 leaks the existence of the vault_id to the server. This is already leaked by every push/pull today. No new exposure.

### 5.5 Check class structure

Each check is a Type_Safe class. The doctor is composed of an ordered list, exactly mirroring the pattern in `sgit_ai/workflow/Workflow.py`:

```python
# sgit_ai/cli/doctor/Check__DNS_Resolve.py
class Check__DNS_Resolve(Type_Safe):
    timeout_seconds : Safe_UInt__Vault_Version = None
    def execute(self, ctx: 'Doctor__Context') -> Schema__Doctor__Check: ...

# Other checks follow identical shape:
#   Check__Parse_URL
#   Check__TCP_Reachable
#   Check__Loopback_Container_Warn
#   Check__TLS_Handshake
#   Check__API_Info
#   Check__Token_Verify
#   Check__Vault_Known
#   Check__Write_Probe
```

The orchestrator `CLI__Doctor` walks the list, short-circuits on FAIL of checks 1-3 (you cannot TLS-handshake against an unreachable host), continues through warnings.

### 5.6 New server endpoints (optional, additive)

Both endpoints are zero-knowledge-safe:

| Endpoint | Method | Auth | Returns | Privacy |
|---|---|---|---|---|
| `/api/info` | GET | none | `{service: 'sgraph-vault', version: 'X.Y.Z', features: ['batch','presigned',...]}` | No vault data |
| `/api/auth/whoami` | GET | access token header | `{scopes: ['read','write']}` (no email, no user identity) | No PII; scope list only |

If the server returns 404 for either, the doctor degrades (WARN, not FAIL). This matters for the migration use case where Dinis runs an older self-hosted SG/API.

## 6. Contract — `sgit vault migrate`

### 6.1 The two migrations

| Mode | When to use | Re-key? | Vault_id changes? | Recommended path |
|---|---|---|---|---|
| **`vault rekey-move`** (existing `Workflow__Vault_Move`) | The source server is suspected compromised; all old clones must be invalidated. | YES | YES | Existing 8-step workflow, renamed. |
| **`vault migrate`** (NEW) | Both servers are trusted; you are simply changing where the ciphertext lives. | NO | NO | New `Workflow__Vault_Migrate` — stream ciphertext as-is. |

### 6.2 Command surface

```
sgit vault migrate [<directory>] --from <source-url> --to <target-url>
                                  [--from-token <T>] [--to-token <T>]
                                  [--yes] [--dry-run] [--resume]
                                  [--no-delete-source]
```

- `<directory>` defaults to `.` (current vault).
- `--from` defaults to the current default remote URL of the vault. The user must spell out `--to`.
- `--from-token` / `--to-token` default to whatever is in the encrypted token store for each URL.
- `--dry-run` runs all checks and computes the transfer set without sending anything to `--to`.
- `--resume` picks up a partially-completed migration from `local/migrate_state.json`.
- `--no-delete-source` skips the source deletion step. Default is to delete after verification.

### 6.3 Workflow steps

```python
# sgit_ai/workflow/migrate/Workflow__Vault_Migrate.py
class Workflow__Vault_Migrate(Workflow):
    name    = Safe_Str__Workflow_Name('vault-migrate')
    version = Safe_Str__Semver('1.0.0')
    steps   = [
        Step__Migrate__Validate_Local,        # local clean, vault_id resolved
        Step__Migrate__Doctor_Source,         # `sgit doctor` against --from must PASS
        Step__Migrate__Doctor_Target,         # `sgit doctor` against --to must PASS
        Step__Migrate__Inventory_Source,      # list_files() on source, enumerate object/ref/index/key ids
        Step__Migrate__Plan_Transfer,         # diff source vs. local; identify the transfer set
        Step__Migrate__Stream_Ciphertext,     # for each file_id: GET source, PUT target. NO decrypt. NO re-encrypt.
        Step__Migrate__Verify_Target,         # batch_read each migrated file_id; compare SHA-256 to source's SHA-256
        Step__Migrate__Swap_Default_Remote,   # write `origin` to point at --to, in remotes.json
        Step__Migrate__Smoke_Test_Pull,       # `sgit pull` against new origin; must round-trip
        Step__Migrate__Delete_Source,         # optional; only after smoke-test PASS
    ]
```

### 6.5 Atomicity, ref handoff, failure recovery

This is the hardest part. The four design choices:

| Question | Decision | Rationale |
|---|---|---|
| **(a) Re-encrypt or stream as-is?** | **Stream as-is.** Files on disk are AES-GCM ciphertext + IV + auth tag. Move the bytes; do not touch the key. | Re-encrypting changes object IDs (since they are content hashes) and breaks every clone. Streaming preserves the vault's commit-graph integrity. |
| **(b) Atomic ref handoff vs. lazy migration?** | **Atomic with verification gate.** Refs are the LAST thing transferred. Until refs land, the source is authoritative. After refs land AND verify, the target is authoritative. The CLI's local remote pointer is the cutover atom. | A partial migration where some refs are on the source and some on the target is unrecoverable. Refs-last gives one atomic flip. |
| **(c) Failure recovery?** | **State machine in `.sg_vault/local/migrate_state.json`** (encrypted, like push_state.json). `--resume` reads it. Verification step is idempotent. Stream step is idempotent (PUT is overwrite-OK if SHA matches). | Same shape as existing `Vault__Sync__Push` resumable state. |
| **(d) New server endpoints?** | **None required.** Source: existing `GET /api/vault/read/{vault_id}/{file_id}` (no auth) or `batch_read`. Target: existing `POST /api/vault/batch/{vault_id}` (write with token + write_key). | Stream migration is a CLI orchestration; the server does not need to know it is happening. |

### 6.6 Transfer set computation

The inventory step lists `bare/data/*`, `bare/refs/*`, `bare/indexes/*`, `bare/keys/*` from the source. Objects are content-addressed; refs are mutable. The plan step:

1. Compute `S = source_objects`, `T = target_objects` (an empty set on first migrate).
2. `transfer_data = S_data - T_data` — bytes-for-bytes, content-addressed, idempotent.
3. `transfer_refs = S_refs ∪ S_indexes ∪ S_keys` — always re-write, since refs are mutable; CAS write-if-match against current source value to detect concurrent source mutations during migration.
4. For each ref, the source SHA-256 at inventory-time is recorded in `Schema__Migrate__State`. At handoff, we re-read each ref from the source and abort if any has changed (someone pushed during migration).

### 6.7 Verification

For each `file_id` in `transferred_ids`, fetch from target and compare SHA-256 to source. The CLI never decrypts; verification is byte-level. Mismatch = abort, do not delete source. Note: this is N round-trips; for vaults with 10k objects this is ~3 minutes assuming 50-op batch_read. Acceptable.

### 6.8 Source deletion

Source is only deleted if `--no-delete-source` is absent AND smoke-test (one full pull from target) passes AND human confirms (unless `--yes`). Deletion uses existing `delete_vault()` endpoint with the source write_key.

### 6.9 Acceptance criteria

| ID | Behaviour |
|---|---|
| AC-M1 | `sgit vault migrate --from A --to B` with no flags fails if doctor fails on either side. |
| AC-M2 | `--dry-run` produces a `Schema__Migrate__State` with `inventory_count > 0`, `transferred_ids == []`, no side effects on either server. |
| AC-M3 | Mid-transfer interruption + `--resume` completes successfully. |
| AC-M4 | If any `verified_ids[i]` SHA-256 mismatches, the entire migration aborts; the source is untouched; the target is left in the partial state with `migrate_state.json` flagged failed. |
| AC-M5 | Vault_id is identical on source and target after migration. |
| AC-M6 | The vault's read_key, write_key, and PKI keypair are byte-identical before and after migration. |
| AC-M7 | An old clone on a different machine can switch to the new server by running `sgit vault remote set-url origin <new>` followed by `sgit pull` — no re-clone required. |
| AC-M8 | Source deletion only fires after AC-M3, AC-M4, AC-M6 all pass. |

### 6.10 Zero-knowledge audit

| Observation | Verdict |
|---|---|
| The CLI reads ciphertext from source and writes the same bytes to target. | OK — server cannot learn anything new. |
| The vault_id, file_ids, and sizes are visible to both servers. | Same as today — not a new leak. |
| Source and target both observe the migration timing window. | New: a colluding source+target pair can correlate the move. **Mitigation:** none required at protocol level; this is fundamentally observable. Document in user-facing migrate help. |
| Target sees the order of file_id uploads. | Existing leak on every push; not new. |
| The CLI never decrypts during transfer. | Verified by the contract; verification step uses SHA-256 of ciphertext, not plaintext content_hash. |

**Verdict:** no new leak vectors. The migration is a pure transport-layer operation.

## 7. The "warn-but-proceed" fix (Finding 1)

### 7.1 Contract

Replace `_prompt_remote_setup` interactive flow with `CLI__Remote_Setup` — a Type_Safe class with:

```python
class CLI__Remote_Setup(Type_Safe):
    token_store      : CLI__Token_Store
    credential_store : CLI__Credential_Store
    debug_log        : object = None

    def prompt_and_verify(self, directory: str, base_url: str = None) -> Schema__Remote_Config:
        # 1. Prompt for URL (default DEFAULT_BASE_URL)
        # 2. Prompt for token
        # 3. Build a transient Schema__Remote_Config
        # 4. Run a SUBSET of doctor checks (parse, dns, tcp, tls, api_info, token_verify)
        # 5. If any FAIL: print the friendly diagnostic, sys.exit(1) — do NOT save anything
        # 6. If all PASS: persist to remotes.json (default=true on first save) + token store
        # 7. Return the saved Schema__Remote_Config
```

### 7.2 The verification subset

Doctor checks 1-7 (skip vault_known and write_probe — both can produce noise during first-time setup). If any check FAILs, exit. No save, no fallback. The subset uses the **same** check classes as `sgit doctor`, so there is one source of truth for what "remote is reachable" means.

### 7.3 Acceptance criteria

| ID | Behaviour |
|---|---|
| AC-W1 | `sgit push` with no remote configured + a healthy default URL prompts for token, verifies, saves, and pushes. |
| AC-W2 | `sgit push` with no remote configured + an unreachable URL prints the diagnostic, exits non-zero, **does not save the token**. |
| AC-W3 | The user can re-run `sgit push` after fixing the URL and the second attempt succeeds. |
| AC-W4 | The token never reaches the disk if any verification check fails. |
