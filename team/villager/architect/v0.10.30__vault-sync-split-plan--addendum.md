## Bridge framing (applies to whole brief)

This split is a **bridge to v0.11.x B13** (5-layer restructure), not the
final shape. Every `Vault__Sync__<X>` extraction will later move into
`sgit_ai/core/actions/<x>/` and be reformulated as a `Workflow__<X>` +
step classes.

**Implication:** keep public surfaces narrow, keep tests portable to the
v0.11.x location without reformulation, and don't over-invest in the
`Vault__Sync__<X>` naming since it changes again.

The bridge is still worth doing: it shrinks the 3,032-LOC monolith
immediately, gives us per-sub-class direct tests, and makes B13 a
one-step move per sub-class rather than a single high-risk swap.

## E3 — `Vault__Graph_Walk` (one tightening)

**Type_Safe field declaration for callbacks.** The proposal declares
`on_commit : callable = None`. The Type_Safe rule (CLAUDE.md) bans raw
primitive fields. Verify whether `callable` is acceptable as a
`Type_Safe` field type; if not, use a typed-callable wrapper or
`Optional[Callable[..., None]]` with whatever Type_Safe accepts. Match
the convention already in use in `Vault__Sync` for the existing
`on_progress` parameter.

## E5 — Class-level split (six tightenings)

### 1. Resolve the state-carrier ambiguity

The plan offers three options for `_init_components` ownership without
picking one:
- facade caches `Vault__Components` and passes to sub-classes
- each sub-class re-calls `_init_components`
- shared `_init_components` on `Vault__Sync__Base`

**Recommendation:** the facade builds `Vault__Components` once per public
call and passes it to the sub-class. Avoids re-deriving keys per method
invocation (B12's `clear_kdf_cache` blunts the cost but it's still an
extra round-trip).

```python
class Vault__Sync(Type_Safe):
    crypto : Vault__Crypto
    api    : Vault__API

    def commit(self, directory: str, message: str = '') -> dict:
        components = self._init_components(directory)  # ONCE per public call
        return Vault__Sync__Commit(crypto=self.crypto,
                                   api=self.api,
                                   components=components).commit(directory, message)
```

`_init_components` lives on `Vault__Sync__Base` so both the facade and
sub-classes can call it when needed.

### 2. Split `Vault__Sync__Admin` further

Twelve+ methods spanning lifecycle, branching, GC, change-packs, remotes,
and uninit/restore is too broad. Replace the single `Admin` sub-class
with three:

| Sub-class | Methods |
|---|---|
| `Vault__Sync__Lifecycle` | `delete_on_remote`, `rekey*` × 5, `probe_token`, `uninit`, `restore_from_backup` |
| `Vault__Sync__Branch_Ops` | `branches`, `merge_abort`, `remote_*` |
| `Vault__Sync__GC` | `gc_drain`, `create_change_pack`, `_auto_gc_drain` |

If splitting raises commit count too much, accept `Admin` as a
miscellany bucket but document explicitly that v0.11.x B13 will
sub-divide.

### 3. Preserve AppSec wiring

When key-handling methods move to `Vault__Sync__Lifecycle` (or `Admin`),
the `clear_kdf_cache` calls wired in by B12 must come with them. Wired
sites today:
- `delete_on_remote` — after `api.delete_vault()`
- `rekey_wipe` — after `secure_rmtree`
- `probe_token` — at every exit path

**Per-extraction acceptance:** after each sub-class extraction touching
key-handling methods, re-run B12's mutation M5 closer test
(`tests/unit/crypto/test_Vault__Crypto__KDF_Cache.py`) and B11's
secure-unlink tests. Fail-closed if either regresses.

### 4. Relax the "no more than 2 files per commit" rule

Current rule (main plan §3) is too strict. E5 sub-class commits will
move 250–400 LOC of source plus their tests; if a sub-class touches
multiple test files (likely — CLI, sync, and crypto tests all hit
`Vault__Sync` methods), the 2-file rule breaks immediately.

**Replace with:** "one logical concern per commit, atomic, suite green."
Intent (small revertible commits) survives without an artificial
file-count limit.

### 5. Make new direct sub-class tests a per-extraction criterion

Main plan §5 says "extraction + new sub-class tests expected to close
the coverage gap during the sprint" — but per-extraction acceptance §4
only requires "coverage ≥ 88%". To actually close the 374-missed-line
gap in `Vault__Sync.py`, each sub-class extraction should add direct
tests against the new class.

**Add per-extraction criterion:** "≥ 5 new direct tests per
`Vault__Sync__<X>` sub-class added in the same commit." Adjust the
number if 5 is too coarse for small sub-classes (e.g., `__GC` may
warrant only 3 — fine, but the test addition is non-negotiable).

### 6. Compatibility-with-v0.11.x checklist

Add the following to the main plan's §5 Dinis review gate (or to this
addendum as a per-extraction sign-off):

- [ ] Plan references v0.11.x design D6 (5-layer architecture) as the
      eventual landing zone.
- [ ] `Vault__Graph_Walk` (E3) public surface kept narrow for future
      Workflow composition.
- [ ] `Vault__Object_Store.batch_download` (E4) accepts `api` per call
      (storage layer purity).
- [ ] `Vault__Sync__<X>` naming acknowledged as a bridge — eventually
      replaced by `core/actions/<x>/Workflow__<X>` + step classes.
- [ ] Each extraction's tests are portable to the v0.11.x location
      without reformulation.

- E1, E2, E4 are clear-cut as written.

- Roll-back protocol (one commit per extraction; revert via `git revert`) stands.
- The 88% coverage acceptance is fine; new direct tests close the gap during the sprint.
