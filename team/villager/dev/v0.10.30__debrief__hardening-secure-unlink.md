## What Was Done

### Helpers Added (2 methods on `Vault__Storage`)

| Location | Method | Purpose |
|---|---|---|
| `sgit_ai/sync/Vault__Storage.py:112` | `secure_unlink(path)` | Overwrite file with zero bytes + fsync, then `os.unlink` |
| `sgit_ai/sync/Vault__Storage.py:139` | `secure_rmtree(directory)` | Walk tree bottom-up, `secure_unlink` every file, `os.rmdir` empty dirs; returns file count |

Design choice: zero bytes (not `os.urandom`) are written because the goal
is to destroy the *key-material pattern* before the kernel reclaims the
blocks ‚Äî not to satisfy a DoD multi-pass standard.  Zero-write is faster
on large files with no effective security trade-off at this layer.

### Call Sites Refactored (1)

| File | Method | Before | After |
|---|---|---|---|
| `sgit_ai/sync/Vault__Sync.py:1798` | `rekey_wipe` | `shutil.rmtree(sg_dir)` | `storage.secure_rmtree(sg_dir)` |

`shutil.rmtree` was an inode-only deletion; blocks remained on disk until
OS overwrite.  With `secure_rmtree`, every file under `.sg_vault/` ‚Äî
including `local/vault_key`, `local/config.json`, and all bare objects ‚Äî
is zero-wiped before its inode is released.

## Disk-Read Assertion Approach

The brief asks how "underlying bytes are not recoverable" is tested.
Kernel-level raw-block reads are not feasible from userspace in a tmpfs
environment, so the assertion approach is:

1. **File gone:** `not os.path.exists(path)` ‚Äî confirmed by every test.
2. **Re-open raises:** `open(path, 'rb')` raises `FileNotFoundError` ‚Äî verified in
   `test_secure_unlink_content_overwritten_before_removal`.
3. **White-box:** The implementation is read directly (no mocking); the
   `open(path, 'r+b') ‚Üí write zeros ‚Üí fsync ‚Üí unlink` sequence is
   observable in source.

This matches the strategy documented in AppSec finding F02 ¬ß8: the test
framework cannot mount a raw-block read on a tmpfs, so the assertion is
correctness-by-inspection plus the "file gone and unreadable" check.

## Mid-Rekey Abort Finding

During implementation the `Test_Vault__Sync__Rekey__Mid_Abort` class exposed
that the `status()` method returns `added` (not `untracked`) for files that
exist in the working tree but not in the HEAD commit.  The test was written
with `untracked` and failed; it was corrected to `added`.

The state-machine analysis (from AppSec finding F02 ¬ß4) confirms:

- **Wipe-only (crash between wipe and init):** `.sg_vault/` is absent, working
  files survive.  Recovery: `sgit init --allow-nonempty` or `sgit rekey init`.
  No data loss.
- **Wipe + init (crash between init and commit):** `.sg_vault/` present with a
  fresh key; working files exist but are uncommitted.  Status shows them as
  `added`.  Recovery: `sgit commit -m rekey` or `sgit rekey commit`.

No redesign is needed ‚Äî the existing two-step recovery path is correct.
The Architect was not escalated.

## Residual Risk Documented

- **SSD TRIM:** Physical blocks may be reallocated by the firmware
  independently of the zero-write.  The helper provides best-effort
  overwrite at the filesystem layer; TRIM-level guarantees require hardware
  or OS-level Secure Erase which is not addressable from userspace Python.
  Documented in `team/villager/appsec/v0.10.30/F02__rekey-key-material.md`.
- **CoW filesystems (btrfs, APFS, ZFS):** Copy-on-Write semantics mean the
  original block may be preserved as a snapshot.  Same userspace limitation.
- **`shutil.rmtree` in `uninit()`:** Line 2953 of `Vault__Sync.py` still
  uses `shutil.rmtree` for the backup-zip workflow.  This is intentional ‚Äî
  `uninit` creates a zip backup before wiping, and the `.sg_vault/` being
  removed there is not key material removal (the user is explicitly keeping a
  backup copy).  Not in scope for this brief.
