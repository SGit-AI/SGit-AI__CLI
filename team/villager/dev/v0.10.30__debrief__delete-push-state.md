# Debrief — Brief 14: delete-on-remote clears push_state.json

**Date:** 2026-05-01
**Brief:** `team/villager/dev/v0.10.30__brief-pack/14__delete-on-remote-clears-push-state.md`
**Architect finding closed:** 06 (resumable-push-state § 2d)

---

## What was done

Fixed a correctness bug where `delete_on_remote` left `push_state.json`
intact on disk. After deleting a vault from the server, stale blob IDs in
the checkpoint file could cause a subsequent push to skip Phase-A blob
uploads — producing dangling references on the server (blobs referenced
by commits/trees that no longer exist there).

---

## Root-cause clarification

`push_state.json` is written during non-first resumable pushes (per-blob
checkpoint) and cleared after a successful push. It persists only when a
push is interrupted mid-Phase-A. `delete_on_remote` only clears the server,
so a stale checkpoint from an interrupted push survived the delete.

On re-push after delete, `_is_first_push()` correctly returns True (server
empty), so the checkpoint would not be consulted — but that relied on the
server being empty. If anything else had written files to the same vault_id
before the re-push (e.g. another client), `_is_first_push()` would return
False and the stale checkpoint would skip blob uploads incorrectly.

The fix removes the ambiguity entirely: delete_on_remote now always clears
the checkpoint.

## Test design note

`push_state.json` is only written by the push loop when `first_push=False`
AND blobs are being uploaded, and is cleared on success. It therefore
cannot be created naturally by a single successful push in a test. The tests
simulate the stale file by writing it directly (a dict with `vault_id`,
`clone_commit_id`, `blobs_uploaded`) — this is the exact format that an
interrupted non-first push would produce and is a more direct and reliable
representation of the bug precondition than trying to interrupt the push
mid-flight.
