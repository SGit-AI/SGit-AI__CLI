### Detailed Changes by Area

#### 1. Self-Describing File ID Convention

Every file ID in the system now encodes its type, derivation method, and mutability:

```
obj-cas-imm-{sha256[:12]}     content-addressed objects (blobs, trees, commits)
ref-pid-muw-{hmac[:12]}       HEAD ref (pseudo-deterministic, multi-writer)
ref-pid-snw-{hmac[:12]}       clone branch refs (single-writer)
idx-pid-muw-{hmac[:12]}       branch index
key-rnd-imm-{random[:16]}     key objects (random, immutable)
```

**Files changed:** `Vault__Crypto.py` (derive_keys, compute_object_id, derive_branch_ref_file_id), all 5 `Safe_Str__*_Id.py` regex validators, `Vault__Key_Manager.py`, `Vault__Branch_Manager.py`, `Vault__Components.py`, `CLI__Vault.py`.

#### 3. Sub-Tree Model (`Vault__Sub_Tree` — new module)

Replaced the flat tree model (one tree with all file paths) with one tree object per directory level:

```
root tree
  ├── readme.md       → blob_id: obj-cas-imm-4f5365b013a0
  └── docs/           → tree_id: obj-cas-imm-ca66507e0204
        └── guide.txt → blob_id: obj-cas-imm-fa04e253fbd9
```

Three core methods:

- **`build(directory, file_map, read_key, old_flat_entries)`** — bottom-up builder. Groups files by directory, builds deepest trees first, stores each as an encrypted object, references child trees from parent trees. Returns root tree ID.
- **`flatten(tree_id, read_key)`** — recursive walker. Returns `{path: dict(blob_id, size, content_hash)}` for comparison against working directory. Decrypts `name_enc`, `size_enc`, `content_hash_enc` from stored entries.
- **`checkout(directory, tree_id, read_key)`** — recursive file extraction. Walks sub-trees, decrypts blobs, writes to working directory.

#### 4. Vault__Commit Simplification

- `create_commit()` now takes `tree_id` (pre-stored tree) instead of a `Schema__Object_Tree` object
- Added `message_enc` support (encrypted commit message)
- Removed `encrypt_tree_entry_fields()` / `decrypt_tree_entry_fields()` — encryption now happens in `Vault__Sub_Tree.build()`
- Added `store_tree()` helper for the sub-tree builder

#### 5. API: Batch Read

- `Vault__API.batch_read(vault_id, file_ids)` — batch read multiple files in one request
- `Vault__API__In_Memory` — added `read` operation support in `batch()` + `batch_read()` method

#### 6. Push: Recursive Tree Upload

`Vault__Batch.build_push_operations()` now:
- Recursively collects all tree objects (root + sub-trees) via `_collect_tree_objects()`
- Deduplicates uploads via `uploaded_ids` set
- Removed browser dual-write

#### 7. Vault__Bare: Sub-Tree Support

Rewritten to use `Vault__Sub_Tree` for:
- `checkout()` — recursive extraction via `sub_tree.checkout()`
- `list_files()` — flattens sub-trees via `sub_tree.flatten()`
- `read_file()` — resolves nested paths via flatten, then reads blob
- `is_bare()` — checks for `bare/refs/ref-pid-*` instead of legacy `refs/head`

#### 8. Vault__Inspector: Sub-Tree Flatten

`inspect_tree()` now uses `Vault__Sub_Tree.flatten()` to resolve nested tree entries into flat paths with correct names (previously showed `None` for sub-tree entries).

#### 9. Merge: Works with Flat Dicts

`Vault__Merge.three_way_merge()` updated in the pull flow — receives pre-flattened `{path: dict}` maps instead of `Schema__Object_Tree` objects. The merge algorithm itself is unchanged (same 14 cases), just operates on dicts.

## What Remains (Not Yet Implemented)

### 1. Remove Plaintext Fields from `Schema__Object_Tree_Entry` (Phase 1.1)

**Status:** Started, not complete. The schema has been updated to remove `path`, `name`, `size`, `content_hash` — but the cascading changes through all consumers are partially done.

**What the brief says:** The stored tree entry should contain ONLY `blob_id`, `tree_id`, `name_enc`, `size_enc`, `content_hash_enc`. No plaintext fields should be serialized to disk.

**What's done:**
- `Schema__Object_Tree_Entry` updated (plaintext fields removed from schema)
- `Vault__Sub_Tree.build()` already encrypts all metadata inline
- `Vault__Sub_Tree.flatten()` already returns plain dicts (not schema objects)

**What's left to wire up:**
- `Vault__Sync.status()` — needs dict-style access (`.get('content_hash')` not `.content_hash`)
- `Vault__Sync._generate_commit_message()` — partially done
- `Vault__Merge.three_way_merge()` — rewritten to accept dicts, but pull flow wiring incomplete
- `Vault__Bare.list_files()` — needs dict access for flatten results
- `Vault__Inspector.inspect_tree()` — needs dict access for flatten results
- Push flow `clone_flat.values()` — returns dicts not entries, `entry.blob_id` → `entry.get('blob_id')`
- All test files that construct `Schema__Object_Tree_Entry` with `path=`, `name=`, `size=`, `content_hash=`
- `Vault__Sub_Tree._store_tree()` — currently creates `Vault__Commit` instance to store trees; needs to serialize directly since entries no longer have the old fields

**Risk:** This is the most cascading change. Every file that constructs or reads tree entries is affected. Estimated: ~2 hours of mechanical wiring + test fixes.

**Note:** There is an in-progress uncommitted change set on the local branch (8 files changed) that begins this wiring but is not yet testable. It should NOT be pushed — it needs to be completed first.

### 2. Remove Legacy Fields from `Schema__Object_Commit` (Phase 1.2)

**Status:** Not started.

**What the brief says:** Remove `parent` (single), `timestamp` (ISO string), `message` (plaintext), `version` (integer). Keep only `parents` (list), `timestamp_ms`, `message_enc`, `schema`.

**What's done:** `message_enc` field was added alongside `message`. `create_commit()` populates `message_enc`.

**What's left:**
- Remove `parent`, `timestamp`, `message`, `version` from schema
- Update `Vault__Commit.create_commit()` to stop setting legacy fields
- Update `Vault__Fetch.find_lca()` — reads `commit.parents` and `commit.parent`
- Update `Vault__Inspector` — reads `commit.message`, `commit.version`, `commit.timestamp`
- Update all tests that assert on legacy commit fields

**Risk:** Medium. Fewer consumers than tree entries, but `parent` vs `parents` handling in LCA is critical.

### 3. Named Branch Merge Commit on Push (Phase 7.3)

**Status:** Not started.

**What the brief says:** After pushing clone changes, create a merge commit on the named branch (signed with the named branch key) so the named branch history shows the merge:

```
named:  C1 ← C2 ← M (merge commit, tree = clone HEAD tree)
                   ↑
clone:  C1 ← C2 ← C3 (pushed changes)
```

**What's needed:**
- In `Vault__Sync.push()`, after updating the clone ref, create a merge commit on the named branch
- Sign with named branch private key (loaded from vault)
- Compare-and-swap update named ref via batch with `write-if-match`
- This is ~30 lines of code in `Vault__Sync.push()`

**Risk:** Low — self-contained addition. But needs integration testing with multi-clone scenarios.

### 4. Graph-Walking Clone (Phase 8.1)

**Status:** Not started.

**What the brief says:** Replace the current `list_files()` + bulk download approach with deterministic graph-following:

1. Read HEAD ref (deterministic file ID — 1 request)
2. Walk commit → tree → sub-trees → blobs (graph following)
3. Read branch index (deterministic file ID — 1 request)
4. Fetch keys referenced by branch index

This eliminates the need for `list_files()` entirely and enables clone to work with any storage backend that supports read-by-ID.

**What's needed:**
- Rewrite `Vault__Sync.clone()` to use the deterministic `ref_file_id` and `branch_index_file_id` from `derive_keys()`
- Add `_fetch_tree_recursive()` helper that follows tree → sub-tree → blob references
- Use `batch_read()` for efficient multi-object fetch

**Risk:** Medium. The clone flow is complex (branch creation, key generation, ref registration). But the graph-walking part is well-defined since all IDs are now deterministic.

### 5. `build_from_flat()` Method on `Vault__Sub_Tree`

**Status:** Referenced by the pull merge flow rewrite (in uncommitted changes) but not yet implemented.

**What it does:** Takes a flat `{path: dict(blob_id, ...)}` map (the merge result) and builds sub-tree objects from it. Essentially the inverse of the working directory scan — it re-encrypts metadata and creates nested tree objects from a flat map where the blobs already exist.

**What's needed:** ~40 lines in `Vault__Sub_Tree`. Similar to `build()` but reads blob_id/content_hash from the dict instead of reading files from disk.
