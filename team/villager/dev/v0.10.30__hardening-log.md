## Brief 11 ‚Äî Secure-unlink for rekey wipe (2026-05-01)

**AppSec finding closed:** F02 (rekey key-material disk residency ‚Äî MEDIUM).

Added `secure_unlink(path)` and `secure_rmtree(directory)` methods to
`Vault__Storage` (`sgit_ai/sync/Vault__Storage.py:112‚Äì162`).
`secure_unlink` opens the file for writing, overwrites its content with
zero bytes, calls `os.fsync`, then `os.unlink`.  `secure_rmtree` walks
the tree bottom-up using `os.walk(topdown=False)`, calling
`secure_unlink` on every file before `os.rmdir`-ing the now-empty
directories.  Zero bytes are used (not `os.urandom`) because the goal
is to destroy key-material patterns, not to pass a DoD wipe standard;
this is functionally equivalent and faster on large files.

`Vault__Sync.rekey_wipe` now calls `storage.secure_rmtree(sg_dir)` in
place of the previous `shutil.rmtree` ‚Äî one call site refactored.

Residual risk on SSDs: TRIM may independently reallocate the physical
blocks; this is not addressable from userspace and is documented in
AppSec finding F02.

New test file: `tests/unit/sync/test_Vault__Sync__Secure_Unlink.py` ‚Äî
19 tests across four classes: `Test_Vault__Storage__Secure_Unlink` (6
tests: empty, small, multi-MB, non-existent noop, FileNotFoundError
assert, content-overwritten assertion); `Test_Vault__Storage__Secure_Rmtree`
(5 tests: non-existent dir returns 0, single file, nested 4 files, empty
dir, deep nesting); `Test_Vault__Sync__Rekey_Wipe__Secure` (6 tests:
`.sg_vault/` removed, `vault_key` gone and unreadable, new vault_key
differs from old, working files intact, objects_removed count,
idempotent double-wipe); `Test_Vault__Sync__Rekey__Mid_Abort` (2 tests:
after wipe-only state is deterministic, wipe+init without commit shows
`added` files in status).

Suite result: **2145 passed** (19 new tests).  Coverage unchanged at

## Brief 12 ‚Äî clear_kdf_cache for PBKDF2 LRU residency (2026-05-01)

**AppSec findings closed:** F03 (PBKDF2 LRU passphrase residency ‚Äî MEDIUM CLI / HIGH agent).

Added `clear_kdf_cache()` method to `Vault__Crypto` (`sgit_ai/crypto/Vault__Crypto.py:185‚Äì199`).  The method calls `_pbkdf2_cached.cache_clear()` on the module-level `functools.lru_cache(maxsize=256)` that caches PBKDF2-SHA256 derivations.  For one-shot CLI runs the cache is harmless; for long-running agent processes the cache would retain passphrase bytes and derived key material indefinitely.  Clearing at natural passphrase-boundary points closes the residency window.

Wired into three locations in `Vault__Sync`:

| Method | When cleared |
|---|---|
| `delete_on_remote` | After `api.delete_vault()` returns ‚Äî vault is gone, key is stale |
| `rekey_wipe` | After `secure_rmtree` ‚Äî old vault material is wiped, old key is stale |
| `probe_token` | After each return path (success vault, success share, failure) ‚Äî probe must leave zero key material in process state |

New test file: `tests/unit/crypto/test_Vault__Crypto__KDF_Cache.py` ‚Äî 3 tests: `test_clear_kdf_cache__functional` (clear empties cache, subsequent derivations still produce correct key), `test_clear_kdf_cache__currsize_zero_after_clear` (cache_info().currsize == 0 immediately after clear), `test_pbkdf2_cache_size_bounded` (M5 mutation closer ‚Äî asserts maxsize==256 and currsize grows, both fail if maxsize=0).

Mutation matrix M5 row updated from "U" to "D".  Suite result: all 3 new tests pass.  No behaviour change ‚Äî the KDF itself is untouched.

## Brief 13 ‚Äî Defensive read-only guard on write_file (2026-05-01)

**AppSec findings closed:** F08 (surgical write path) ‚Äî partial. **Dev finding closed:** 8.6 (error-handling consistency on write path). **Architect finding closed:** 02 (CLI/vault boundary).

Added a fail-closed guard at the top of `Vault__Sync.write_file` (`sgit_ai/sync/Vault__Sync.py:246`): if `c.write_key` is absent or empty, `Vault__Read_Only_Error` is raised before any I/O.  A read-only clone (one that stores only a `read_key` in `clone_mode.json`) therefore cannot silently accept writes.

`_init_components` was simultaneously hardened: if `clone_mode.json` exists but cannot be JSON-parsed, or if it is `mode=read-only` without the required `read_key`/`vault_id` fields, it raises `Vault__Clone_Mode_Corrupt_Error` (fail-closed). Previously it fell through to full-mode treatment, granting write access to what might be a read-only clone.

Both exception classes live in `sgit_ai/sync/Vault__Errors.py` with module-level message constants (`MSG_WRITE_KEY_MISSING`, `MSG_CLONE_MODE_CORRUPT`) so call sites are grep-able and the message text is testable without raw string comparisons.

New test file: `tests/unit/sync/test_Vault__Sync__Write_File__Guard.py` ‚Äî 7 tests covering: read-only clone raises `Vault__Read_Only_Error`; invalid-JSON `clone_mode.json` raises `Vault__Clone_Mode_Corrupt_Error`; missing-fields `clone_mode.json` raises `Vault__Clone_Mode_Corrupt_Error`; full-mode vault still writes successfully; error messages are descriptive; no partial disk write occurs when the guard fires.

Mutation matrix M7 updated from "U" to "P" ‚Äî the `clone_mode.json`-corruption sub-path is now guarded; the encryption-skip sub-path (`crypto.encrypt` ‚Üí identity) remains open for brief 20.
