## Existing infrastructure (already in place)

`tests/unit/sync/vault_test_env.py` — `Vault__Test_Env` class with
`setup_single_vault` + `setup_two_clones`. Class-level snapshot →
`copytree` per test. Saves **~400 ms → ~3 ms per test** vs full vault init.

`tests/unit/sync/conftest.py` — five fixtures from v0.10.30 design D03:

| Fixture | Scope | Covers |
|---|---|---|
| F3 `bare_vault_snapshot` | module | `small_vault`, `read_list_vault` variants — bare layouts only |
| F4 `bare_vault_workspace` | function (factory) | per-test `copytree` of an F3 variant |
| F5 `probe_vault_env` | session | one shared probe-test vault |
| F6 `simple_token_origin_pushed` | module | post-push origin for simple-token tests (2 consumers) |

`tests/unit/cli/conftest.py` — `pki_keypair_snapshot` (F1), `pki_workdir` (F2).

**This is good infrastructure.** The problem is **adoption**, not
architecture.

## Five new fixture types worth adding

Today's fixtures cover bare vaults + probe + simple-token. Tests that
need richer pre-built state still bootstrap from scratch:

| Proposed fixture | What it pre-builds | Consumers |
|---|---|---|
| `two_clones_pushed` | Alice init+commit+push, Bob clone | `Multi_Clone` family, `Branch_Switch`, merge tests |
| `vault_with_N_commits` (factory: N = 1, 5, 20) | Linear commit history | diff, log, blame, inspect tests |
| `vault_with_pending_changes` | Vault + working-copy modifications staged | status, add, reset, stash tests |
| `vault_with_branches` | Two branches diverged from a common base | branch ops, merge tests |
| `read_only_clone` | A clone in read-only mode (clone_mode.json with read_key only) | write-guard tests, F08 / B13 work |

All are session- or module-scoped and use the same snapshot+copytree
isolation pattern as F3/F4. Each new fixture is small (~50 LOC) and
slots into the existing `tests/unit/sync/conftest.py`.

## Encryption redundancy patterns to remove

These are tests where encryption is **incidental** — the test isn't
testing encryption, but full encryption is run as part of setup. Each
case is solvable with a pre-built fixture or pre-derived constant:

1. **Diff / log / blame tests doing init+commit+commit just to get
   two commits**. Could use `vault_with_N_commits(2)` — same end
   state, ~5 ms instead of ~800 ms.
2. **Inspect tests building a vault to inspect**. Same pattern.
3. **CLI command tests that init a vault to test a downstream command**
   (e.g., `test_CLI__Commands.py`'s 9 raw setups). Most could use F3/F4
   or `Vault__Test_Env`.
4. **AppSec tests that derive keys from passphrase to test something
   non-derivation**. Could pre-derive a known test vault_key once at
   session scope and expose `read_key_bytes` / `write_key_bytes` /
   `vault_id` directly.
5. **Object-store tests building blobs to test load/save paths**.
   A session-scoped `precomputed_encrypted_blobs` fixture (a dict of
   `{name: ciphertext_bytes}`) eliminates per-test encryption when the
   test is about disk I/O, not crypto.

The "no mocks" rule is preserved — these are all real objects, just
**built once and shared with copy semantics** rather than rebuilt per test.

## Pre-derived key cache

PBKDF2 with 600k iterations costs ~50–100 ms per derivation. The
existing `lru_cache(maxsize=256)` on `_pbkdf2_cached` blunts this within
a process — but **only for repeat calls with the same passphrase**.
Tests use unique passphrases by convention (each test class picks its
own), so cache hit rate is low across tests.

Proposal: a session-scoped fixture exposing a small set of pre-derived
keys for common test vault_keys:

```python
@pytest.fixture(scope='session')
def known_test_keys():
    return {
        'coral-equal-1234'  : derive_keys_from_vault_key('coral-equal-1234'),
        'give-foul-8361'    : derive_keys_from_vault_key('give-foul-8361'),
        'azure-hat-7-9912'  : derive_keys_from_vault_key('azure-hat-7-9912'),
        # …extensible
    }
```

Tests that need `read_key_bytes` for a known vault_key consume this
instead of re-deriving. Saves ~50–100 ms per consumer × ~80 consumer
tests = ~4–8 seconds suite-wide.

## Centralisation: move `Vault__Test_Env` to a shared location

Today `Vault__Test_Env` lives in `tests/unit/sync/vault_test_env.py`.
Tests in `api/`, `cli/`, `objects/`, `transfer/`, `appsec/` would
benefit from it but currently don't import it (cross-namespace imports
are awkward in test code).

Proposal: move to `tests/_helpers/vault_test_env.py` (or a top-level
`tests/conftest.py` exposing it). The five new fixtures from §"Five
new fixture types" go there too. `tests/unit/sync/conftest.py` then
becomes thin — just sync-only specifics.

## Coverage gaps (path to high-90s%)

Current: 86% (baseline) / 88% (per B22 plan). Largest single pocket:
**`Vault__Sync.py` at 81%, with 374 missed lines** — accounts for
roughly half of all uncovered lines.

Next-largest pockets (per QA baseline):
- `Vault__Backend__API.py` — 0% (21 LOC)
- `Vault__API.py` — 33.9% (B18 closed most; verify post-B18)
- `API__Transfer.py` — 46.5% (B18 closed most; verify)
- `Vault__Diff.py` — 61.4% (B18 added coverage; verify)
- `CLI__Diff.py` — 62.0%

Path to 95%+:
1. **B22 sub-class direct tests** (per the addendum): each
   `Vault__Sync__<X>` extraction lands ≥ 5 direct tests. ~10 sub-classes
   × 5 tests = 50 new direct tests. Should close ~150–200 of the 374
   missed `Vault__Sync.py` lines.
2. **Long-tail error paths.** Many sync methods have `except Exception:`
   blocks that no test exercises. A targeted brief to enumerate and
   cover them = another 50–100 lines closed.
3. **CLI handler coverage.** CLI files run through CLI tests but rarely
   have direct handler tests. Add a thin direct-handler test per
   command = quick coverage win.
4. **Plugins (when v0.11.x B14 lands).** Each plugin gets its own
   tests/ subdir and direct coverage. Adopting the layered architecture
   makes per-plugin coverage targets natural.

Realistic milestone: **≥ 92% by end of B22**, **≥ 95% by mid-v0.11.x**.

## Proposed brief: B23 — Test Framework Round 2

Naming follows v0.10.30 brief-pack convention. Owner: **Villager Dev**
(primary) + **Villager DevOps** (timing measurement). Effort: **~3–4
days**.

Scope:

1. **Adoption pass** — refactor the ~16 sync test files (and 2 cli
   files) listed in §"Adoption gap" to use F3–F6 or `Vault__Test_Env`.
   Target: per-file warm runtime drops by ~40%.
2. **Five new fixtures** (§"Five new fixture types") added to a
   shared `tests/_helpers/` or root `conftest.py`. Each fixture has its
   own tests for build correctness + isolation.
3. **`known_test_keys` session fixture** + adoption in tests that
   need pre-derived keys.
4. **Move `Vault__Test_Env` to `tests/_helpers/`**; expose to all test
   subdirs; deprecate the sync-only path with a one-line forwarding
   import for one release.
5. **Encryption-redundancy audit doc** — small companion doc listing
   tests that still do incidental encryption after this brief, for
   future work.

Acceptance:
- All 2,367+ tests still pass.
- Coverage ≥ 88% (no regression).
- Suite warm runtime drops by ≥ 25% on whichever runner is the
  reference (likely 285s → ~210s on the slow runner; 124s → ~90s on the
  baseline machine).
- Mock-pattern line count unchanged or lower.
- No new `__init__.py` under `tests/`.
- Each new fixture has its own round-trip + isolation test.

Out of scope:
- Adding new test scenarios beyond fixture refactors.
- Coverage push beyond what B22 sub-class tests will close (separate
  brief if 95% target is to be met before v0.11.x).
