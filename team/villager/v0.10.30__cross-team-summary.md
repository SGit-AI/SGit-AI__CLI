## 1. Headline verdict

**No P0 findings. No SEND-BACK-TO-EXPLORER findings.** The Sonnet team's
v0.10.30 sprint respected the Explorer-defined architectural boundaries,
preserved the `bare/*` vault format, and shipped cryptography that — by

## 2. The three issues all three angles independently surfaced

These appeared in every consolidated report. They are the highest-signal
findings of the review.

### 2.1 The headline crypto change shipped without direct tests

The HMAC-IV deterministic-encryption change (commit `4d53f79`) is the
biggest cryptographic behaviour change in months. It directly underwrites
the new tree-CAS deduplication claim and the IND-CPA security argument.

| Angle | Finding | Evidence |
|---|---|---|
| Architect | 01 + 08 | `git show 4d53f79 --stat` shows zero test additions; debrief 07's "3 determinism assertions" don't exist |
| Dev | 09 | `encrypt_deterministic` / `encrypt_metadata_deterministic` have zero direct tests across 8 call sites in `Vault__Sub_Tree`; no browser interop vector |
| AppSec | M1, M2 (mutation matrix) | Replacing HMAC with plain SHA-256 in `Vault__Crypto.py:169` would NOT be detected by any current test |

This is the single most important follow-up of the sprint. Five concrete
test cases are listed in Architect finding 01; AppSec mutations M1/M2
specify exact assertions; Dev finding 09 lists the missing test file
(`test_Vault__Crypto__Deterministic.py`).

### 2.2 `Vault__Sync.py` is now 2,986 LOC with visible duplication

The single class hosts the entire sync surface. Twenty-two commits in two
weeks have left structural scars.

| Angle | Finding | Evidence |
|---|---|---|
| Architect | 04 | Two BFS implementations (clone vs pull) with diverging optimisations; two near-identical 90-line `Vault__Sub_Tree.build*` methods; the HMAC-IV change had to be applied to both in lockstep |
| Dev | 06 | File grew ~2400 → 2986 LOC this sprint; 8+ logical responsibilities; eight candidate sub-classes identified |

Both angles agree on a Phase-3/4 refactor: extract `Vault__Graph_Walk`,
unify `build*`, pull blob bucketing into `Vault__Object_Store`. Estimated
~300 LOC removed, every future BFS bug fix becomes a one-line change.

### 2.3 Three new on-disk JSON files have no Type_Safe schema

Slow-burn structural-debt finding raised by all three angles for different
reasons.

| Angle | Finding | Concern |
|---|---|---|
| Architect | 05 | `Schema__Local_Config` declares one field; on-disk file has four (`my_branch_id`, `mode`, `edit_token`, `sparse`). `push_state.json` and `clone_mode.json` have no schema at all. Type_Safe rule violated by implementation, even though no class declares a raw primitive. |
| Dev | 10 | Two new persisted JSON state files (`clone_mode.json`, `push_state.json`) without `Schema__*` classes. Worst offender: `Vault__Sync.write_file` returning raw `dict`. |
| AppSec | M8 (mutation matrix) | A `Schema__Push_State` round-trip invariant would close the M8 mutation gap **structurally** rather than by ad-hoc validation. |

**This is the one fix that touches all three angles in the same way.**
Adding `Schema__Push_State`, `Schema__Clone_Mode`, and extending
`Schema__Local_Config` is a small, well-scoped piece of work that resolves
a code-quality regression (Dev), restores Type_Safe boundary integrity
(Architect), AND tightens an adversarial-test gap (AppSec).

## 3. Bug discovered during review

**`delete-on-remote` followed by `push` (without `rekey` between)** leaves
`push_state.json` in place. The next push will skip Phase-A blob uploads
thinking they're on the server, while Phase-B uploads commits/trees that
point at deleted blobs. Result: server holds dangling references.

- Source: Architect finding 06.
- One-line fix proposed: clear push-state inside `delete_on_remote`.
- Severity: MEDIUM (data integrity; user-visible failure mode).

This wasn't on any agent's plan; it surfaced during code reading.

## 4. Security findings (AppSec-led)

### 4.1 HMAC-IV leakage map (per Dinis's request: "what exactly are we leaking?")

**What the server CAN learn (ciphertext-only):**
- Equality of any two encrypted metadata fields *within a single vault*
  (same `name_enc` ⇒ same plaintext name; ditto for `size_enc`,
  `content_hash_enc`, `content_type_enc`).
- Equality of any two encrypted tree objects within a vault (this is
  intentional — that's what enables CAS dedup).
- Per-commit subtree churn pattern.
- Per-vault structural fingerprint over time.

**What the server CANNOT learn (without `read_key`):**
- Plaintext of any encrypted field — names, sizes, hashes, MIME, bodies.
  AES-GCM IND-CPA is preserved.
- Any key material — `read_key`, `write_key`, `vault_key`, passphrase.
- Cross-vault correlation. Two vaults with the same filename produce
  *different* `name_enc` because their `read_key`s differ.

**Verdict:** Construction is sound. Leakage surface is intra-vault only.
Recommendation: surface this in the user-facing security model so users
know to put highly sensitive small files in separate vaults. No code
change required. Full detail in AppSec F01.

### 4.2 Real fixes worth shipping in a small hardening PR

All small, well-scoped, no behaviour change for normal users.

1. `os.chmod(path, 0o600)` on every save in `.sg_vault/local/` (AppSec F02, F07, F11) — single helper, applied in 3-4 sites.
2. `_secure_unlink(path)` helper that overwrites then unlinks; used in `rekey_wipe` (F02).
3. `Vault__Crypto.clear_kdf_cache()` method + wire into rekey/probe/delete (F03 — addresses module-level `lru_cache(maxsize=256)` retaining old passphrases in agent contexts).
4. Defensive `if not c.write_key: raise` at top of `write_file` (F08).
5. Add `pip-audit` to CI + investigate the env divergence (sandbox runs `cryptography 41.0.7` despite the `>=43.0.0` pin — F12).

### 4.3 Accepted-risk-document-only (per Dinis)

- `clone_mode.json` storing `read_key_hex` (consistent with other `.sg_vault/local/` secrets).
- New `vault_key` printed to stdout during `sgit rekey` (intentional UX).
- HMAC-IV intra-vault structural leakage (acceptable per threat model).
- `delete-on-remote` cannot guarantee server-side ciphertext destruction.
- Sparse-clone access patterns visible to server.
- `probe_token` is a token-existence oracle by design.

## 5. Test-quality findings (Dev + AppSec joint)

### 5.1 Mutation-test coverage gap

AppSec ran a mental pass over 10 adversarial mutations. Predicted detection:

| | Detected | Undetected | Partial |
|---|---:|---:|---:|
| Crypto mutations (M1–M3) | 1 | 2 | 0 |
| Sync-layer mutations (M4–M8) | 2 | 3 | 0 |
| Lifecycle mutations (M9–M10) | 1 | 1 | 1 |
| **Total** | **4** | **6** | **1** |

Most consequential undetected mutation: **M7** — replacing
`crypto.encrypt(read_key, file_content)` with the identity in `write_file`
(`Vault__Sync.py:282`) would NOT be caught by any test. No test reads
`bare/data/{blob_id}` and asserts ciphertext ≠ plaintext for the
write_file path. Full matrix at `team/villager/appsec/v0.10.30/M00__mutation-test-matrix.md`.

### 5.2 Mock count

Dev's `grep` sweep:

- 553 mock-pattern lines total in `tests/`.
- `unittest.mock` / `MagicMock` / `@patch`: 34 lines across 9 files.
- `monkeypatch`: 519 lines across 19 files.
- **Newly introduced this sprint:** 0 new `unittest.mock` imports + 4 homemade `FakeArgs` stubs in `test_CLI__Vault__Write.py`.
- Bulk of the 553 is carryover. Recommend Sherpa reconcile counting method against the v0.5.11 reported "18 violations" baseline — they are not measuring the same thing.

### 5.3 Largest coverage gaps

1. Resumable-push checkpoint (`_load_push_state` / `_save_push_state` / `_clear_push_state` triplet) — no dedicated test file.
2. `encrypt_deterministic` / `encrypt_metadata_deterministic` — zero direct tests.
3. `probe_token` share-token branch — zero coverage.

Suggested new test files (from Dev finding 07):
`test_Vault__Sync__Sparse.py`, `test_Vault__Sync__Push_Checkpoint.py`,
`test_Vault__Crypto__Deterministic.py`.
