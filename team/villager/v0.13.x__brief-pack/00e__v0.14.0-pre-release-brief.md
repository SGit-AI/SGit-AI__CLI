## Items to fix before tagging v0.14.0

### 🔴 1. Clone resolution broad try/except masks mid-clone failures

**File:** `sgit_ai/core/actions/clone/Vault__Sync__Clone.py`, lines 258–270.

**Symptom seen by users:** "Vault cloned but the local key file didn't get written. Trying again into a fresh directory worked."

**Root cause:**

```python
# Step 1: try SGit-AI vault lookup
_p('step', f'Checking SGit-AI for vault: {token_str}')
try:
    keys      = self.crypto.derive_keys_from_simple_token(token_str)
    vault_id  = keys['vault_id']
    index_id  = keys['branch_index_file_id']
    index_fid = f'bare/indexes/{index_id}'
    idx_data  = self.api.batch_read(vault_id, [index_fid])
    if idx_data.get(index_fid):
        _p('step', 'Vault found on SGit-AI — cloning with simple token keys')
        return self._clone_with_keys(token_str, directory, on_progress, sparse=sparse)
except Exception:
    pass
```

The `try` block wraps **both** the probe (lines 261–265) **and** the actual call to `_clone_with_keys` (line 268), which runs the full 10-step clone workflow.

If anything in steps 1–9 raises (transient network blip during `Download_Blobs`, IO hiccup in `Extract_Working_Copy`, anything), the exception is silently swallowed, the code falls through to the SG/Send transfer probe, that fails with `"No vault or transfer found"`, and **the user is left with a partially-populated directory** — `bare/data/` has objects, the working copy may be partly there, but **step 10 (`Setup_Local_Config`) never ran**, so the vault-key file is missing.

On retry into a fresh directory, the transient cause typically doesn't recur and the workflow completes all 10 steps. Hence "worked on the second pass."

**Fix:** narrow the `try` to wrap only the probe; call `_clone_with_keys` outside the try so any mid-clone exception surfaces with a real error message.

```python
try:
    keys      = self.crypto.derive_keys_from_simple_token(token_str)
    vault_id  = keys['vault_id']
    index_id  = keys['branch_index_file_id']
    index_fid = f'bare/indexes/{index_id}'
    idx_data  = self.api.batch_read(vault_id, [index_fid])
    found_on_sgit_ai = bool(idx_data.get(index_fid))
except Exception:
    found_on_sgit_ai = False

if found_on_sgit_ai:
    _p('step', 'Vault found on SGit-AI — cloning with simple token keys')
    return self._clone_with_keys(token_str, directory, on_progress, sparse=sparse)

# Step 2: try SG/Send transfer lookup ...
```

**Test required:**
- `tests/unit/core/actions/clone/test_Vault__Sync__Clone__Resolve.py::test_mid_clone_failure_surfaces_real_error` — install a fake `Vault__API` whose `batch_read` succeeds for the index probe but raises mid-clone (e.g. on the second `batch_read` call). Assert the exception bubbles up with a meaningful message, not `"No vault or transfer found for..."`.

**Effort:** ~15 min.

### 🟠 2. B05 lazy-fetch wiring — still unmet

**Files affected:** `sgit_ai/core/Vault__Sync__Base.py:212-279` (the `fetch_tree_lazy` method exists), and read-path consumers (none yet).

**Brief acceptance criterion** (`team/villager/v0.13.x__brief-pack/brief__05__per-mode-clones-no-pack.md:39-43`):

> Plus a small lazy-fetch path:
> - `Vault__Sync.fetch_tree_lazy(tree_id)` — checks if the tree is in `bare/data/`; if not, downloads it via `api.batch_read([f'bare/data/{tree_id}'])`.
> - **Wire this lazy fetch into `Vault__Sync__Pull._fetch_missing_objects` and into the history-traversal commands** (`history log -p`, `history diff <past>`, etc.).

**Current state:** the method is implemented and exposed via `Vault__Sync.fetch_tree_lazy`, but `grep -rn fetch_tree_lazy sgit_ai/cli sgit_ai/plugins sgit_ai/storage` returns zero hits. Today, after `sgit clone-branch`, running `sgit history log -p` or `sgit history diff <past-commit>` on an older commit will silently fail (object not local) instead of triggering a lazy fetch.

**Decision needed — pick one:**

- **(A) Wire `fetch_tree_lazy` into the read paths.** Cleanest hook is `Vault__Object_Store.load(oid)` — when the local file is missing, look up the parent tree in flight (or fail with a structured "lazy-fetch needed" exception that callers translate to a `fetch_tree_lazy` invocation). Add the brief-required test: clone-branch a vault, run `history log -p`, assert `lazy-fetch.log` was written. **Effort:** ~2 hours.

- **(B) Defer wiring; document `clone-branch` as HEAD-only.** Add a one-liner to the `clone-branch --help` text: `"Note: only HEAD is fully cloned. Run 'sgit fetch <path>' or 'sgit pull' to access older history."` and add a similar note to README. Track wiring as a follow-up brief. **Effort:** ~10 min.

**Recommendation: (A)**, because the brief promised it, and HEAD-only-but-pretends-otherwise is the kind of UX paper-cut that turns into bug reports. But (B) is acceptable if the team is time-constrained — provided the doc change actually lands.

### 🟡 3. B02 migration hardening — 5 items from 00c review

**File:** `sgit_ai/migrations/tree_iv/Migration__Tree_IV_Determinism.py`.

`grep -A2 "except" Migration__Tree_IV_Determinism.py` confirms 5 `try/except: pass` sites are still present. The migration command is user-facing (`sgit migrate apply`) and operates on encrypted vault data. Silently masking corruption as "successfully applied" damages trust hard.

#### 3a. Surface decrypt failures via stats dict (5 sites)

Replace `try / except Exception: pass` with error collection. Sites (line numbers approximate, may shift):

- The `is_applied` 5-tree sample loop — if a tree fails to decrypt, current code keeps sampling. Fix: count decrypt failures and surface them in the return value (`(applied, n_failures)`), or raise immediately on the first failure with a clear message.
- The `_collect_tree_ids` BFS (commit walk) — silently dropping commits with bad decrypt is wrong; collect into a `_corrupt_commits` list and surface in the migration result.
- The `_collect_tree_ids` BFS (tree walk) — same.
- The `_rewrite_trees` rewrite loop — if a tree fails to re-encrypt, abort the migration with a clear error pointing at the corrupt object. Do NOT mark as "applied" while leaving corruption masked.
- The `_rewrite_commits` rewrite loop — same.

In all cases, the `apply()` return value should contain a non-empty `errors` field (currently absent from the stats dict). The runner should surface those errors in `Migration__Runner.apply()` so the user sees them.

#### 3b. Topo-sort cycle handling — escalate, don't paper over

Lines ~215–217 and ~267–269:

```python
# Append any unsorted nodes (shouldn't happen in a valid vault)
for tid in tree_ids:
    if tid not in sorted_trees:
        sorted_trees.append(tid)
```

Replace with:
```python
unsorted = [tid for tid in tree_ids if tid not in sorted_trees]
if unsorted:
    raise RuntimeError(
        f'Vault tree graph contains a cycle or unreachable nodes: {unsorted[:3]}... '
        f'({len(unsorted)} total). Migration aborted to prevent silent corruption.'
    )
```

A corrupted graph should NOT proceed silently and produce subtly-wrong rewritten commits.

#### 3c. Use `Schema__Migration_Record` and `Schema__Migrations_Applied` in the runner

`sgit_ai/migrations/Migration__Runner.py:14-37` — the runner reads/writes raw `dict`s. The schemas exist (`sgit_ai/schemas/`) but aren't used. Type_Safe round-trip invariant is not enforced.

Replace the dict path with `Schema__Migrations_Applied.from_json(data)` / `obj.json()`. Add a round-trip test: write a record, re-read, assert `record.json() == reread.json()`.

#### 3d. Add comment to `is_applied` sampling heuristic

Line 35 — `for tid in list(tree_ids)[:5]: # sample first 5`. Add a docstring/comment explaining: a vault with N trees and 3 random-IV outliers may be falsely marked "applied" by sampling only 5; the migration becomes a no-op for those 3, leaving them suboptimally encrypted. The cost is suboptimal dedup, not corruption.

#### 3e. Branch index sanity check

Confirm that `Schema__Branch_Index` does not contain direct commit-id references that need updating during the tree-IV migration. (Branch-index entries should reference ref files, not commits directly.) If the check confirms no direct commit-id refs: add a one-line comment in `Migration__Tree_IV_Determinism._update_refs` documenting that the branch index doesn't need updating. If commit-id refs DO exist: extend `_update_refs` to walk the branch index too.

**Effort for all of 3a-3e:** ~½ day.

### 🟡 4. B22 — Workflow__Runner exception fallback

**File:** `sgit_ai/workflow/Workflow__Runner.py`, line ~126.

Current:
```python
raise type(_exc)(error_msg) from _exc
```

This breaks for typed exceptions whose constructor doesn't accept a single string (e.g. exceptions with required positional args, or with `__init__` that validates). Add a `TypeError` fallback:

```python
try:
    raise type(_exc)(error_msg) from _exc
except TypeError:
    raise RuntimeError(error_msg) from _exc
```

Add a test that registers a step which raises a multi-arg-constructor exception (e.g. `class MyError(Exception): def __init__(self, code, msg): ...`) and asserts the runner re-raises with a useful message rather than crashing.

**Effort:** ~10 min.
