**Author:** Villager orchestrator (this session)

**Branch:** `claude/villager-multi-agent-setup-sUBO6`

This document captures the strategy for attacking SGit-AI clone
performance, starting from the case-study vault Dinis provided and
moving outward to push / pull / fetch later.

---

## 1. The smoking gun

Real-world clone trace from a 4-agent collaborative website vault:

```
⏱ commits 5.3s   trees 184.2s   blobs 12.6s   checkout 0.1s
   42 commits    2,375 trees    165 blobs
```

**91% of clone time is spent walking trees.** Trees are ~14× more
numerous than blobs. For comparison, blobs (the actual file contents)
take 6% of the time and complete in 12.6s for 165 files.

Tree walking happens at `sgit_ai/sync/Vault__Sync.py:1383-1409`. The
Phase-4 BFS walk seeds the queue with the root tree of **every commit
in history** (line 1372 appends every commit's `tree_id` to
`root_tree_ids`), then walks every subtree. After dedup via the
`visited_trees` set, 2,375 distinct tree objects are downloaded and
decrypted.

For a user who just wants HEAD's working copy, walking 42 commits' worth
of historical trees is wasted work.

---

## 2. Hypotheses for where the 184s actually goes

Before designing fixes, we need to know which of these dominate.
Phase 0 (instrumentation) settles this.

| Hypothesis | Mechanism | How to verify |
|---|---|---|
| **H1** Many small BFS waves with high HTTP latency per wave | Each wave is one `batch_read` HTTP. If the tree depth is high or the per-level branching factor is low, we get many sequential round-trips. | Count waves, plot wave size distribution. |
| **H2** Per-tree decryption + JSON-parse overhead × 2,375 | Python overhead between fast C calls (AES-GCM, JSON) accumulates. | Per-tree wall-clock around `vc.load_tree(...)`. |
| **H3** Pre-HMAC-IV trees fail to dedup | The May-1 deterministic-IV change is recent. Older commits in this vault encrypted trees with random IVs, so identical content yields different tree-ids — `visited_trees` can't dedup them. | Count tree-id frequency across history; ratio of "would-be-dedup'd" trees. |
| **H4** Server response composition is slow per batch | Even with batches, the server may iterate per-object on its side (S3 GET per file in the batch). | Server-side timing of `batch_read` from API logs. |
| **H5** Walking historical trees is fundamentally unnecessary at clone | A normal user wants HEAD, not history. Historical trees are only needed for `log -p`, `diff <past>`, `checkout <past>`. | Architectural; not a measurement. |

**Most likely:** H5 is the lever (skip historical trees entirely),
with H1/H2/H3 as secondary contributors that the case-study vault
amplifies.

1. Visualization / understanding / step-by-step utilities **first**.
2. Then sub-command consolidation (the `inspect-*` / `diff-state`

   migration command (rekey precedent) acceptable.

## 4. Phase 0 — Instrumentation + visualization

The principle: **you cannot optimise what you cannot see.** Before
any production code change, build a small set of CLI tools focused on
making the inside of clone (and later push/pull) visible.

Proposed tools (all read-only against an existing vault — no new
backend behaviour required):

### 4.1 `sgit dev profile clone` — instrumented clone

Runs the existing clone path with detailed instrumentation. Output:

- Per-phase wall-clock (already exists in trace; expand to per-wave).
- Per-wave: wave number, wave size, HTTP wall-clock, decrypt wall-clock,
  JSON-parse wall-clock.
- Per-tree (sampled, top-N slowest): tree-id, decrypt time, parse time,
  entry count.
- HTTP call count (total round-trips).
- Bytes downloaded vs bytes after decrypt.
- Tree-id frequency across history (for H3).

Output formats: text summary + machine-readable JSON for further
analysis.

### 4.2 `sgit dev tree-graph` — visualize the tree DAG

Given a vault, produce:

- A textual representation of the tree DAG (commit → root tree →
  subtrees).
- Per-commit: total trees rooted, unique trees added, dedup ratio.
- Per-depth-level histogram of tree count.
- "If we cloned only HEAD" counterfactual: how many trees would we
  download?

Output: ASCII tree, optionally Graphviz `.dot` file for rendering.

### 4.3 `sgit dev server-objects` — inventory the remote

Counts and classifies objects on the server for a given vault:

- Total object count by type (commit / tree / blob / index / ref / key).
- Reachable from current HEAD vs only reachable from history.
- Largest N objects (post-decrypt size).
- Hot tree-ids (referenced from many commits — would benefit from
  dedup).

### 4.4 `sgit dev step-clone` — step-by-step clone

A clone that pauses between phases and shows what just happened, what
files were created, what's next. Useful for teaching and for
diagnosing stuck clones.

### 4.5 `sgit dev replay` — replay a captured clone trace

Records a clone's trace to JSON; replays it offline (no network). Lets
us diff before/after on identical inputs.

**Constraint:** all Phase-0 tools are **read-only on the network and
read-only on `sgit_ai/` source paths used by main commands.** They may
import from `sgit_ai/` freely but must not modify or replace existing
methods.

## 5. Phase 1 — CLI sub-command consolidation

Today: 67 top-level commands (`grep -c add_parser sgit_ai/cli/CLI__Main.py`
returns 78 entries; ~67 unique top-level command names). Several
clusters cry out for namespacing:

| Existing top-level | Proposed namespace | Notes |
|---|---|---|
| `inspect`, `inspect-object`, `inspect-tree`, `inspect-log`, `inspect-stats` | `sgit inspect <subcommand>` | Already named with prefix; trivial move |
| `diff-state` | `sgit diff state` (or `sgit inspect diff-state`) | Sister of inspect family |
| `cat-object`, `decrypt`, `encrypt` (debug) | `sgit dev <subcommand>` | Developer / debug tools |
| `derive-keys`, `show-key` | `sgit keys <subcommand>` | Or merge into `sgit pki` |
| `dump`, `debug` | `sgit dev <subcommand>` | Same as above |
| (new) profile clone, tree-graph, server-objects, step-clone, replay | `sgit dev <subcommand>` (perf) | Where Phase-0 tools live |

The Phase-0 tools naturally land under `sgit dev <…>`, which is also
where the existing dev-flavoured commands belong. **PKI is already
namespaced** (`sgit pki <…>`), giving us the model.

Important constraints (per Dinis's principles):
- **Backward compat preferred.** For each renamed command, keep the
  old top-level name as a thin alias that prints a deprecation notice
  and forwards. Drop aliases in a future version.
- **No behaviour change.** The consolidation is purely structural;
  the underlying logic is the same.

Phase 1 also covers cleanup of the long-tail top-level commands once
the inspect/dev families are settled. We'll inventory them in a
dedicated brief.

## 7. Phase 3 — Design + implement fixes

Candidate fixes ranked by expected leverage. The actual choice is
made after Phase 2 produces numbers.

### 7.1 Fix A — Lazy historical tree fetch (high leverage, no protocol change)

Default clone walks **only HEAD's tree**. Historical trees are
fetched on demand when the user runs `log -p`, `diff <past>`,
`checkout <past>`. Same pattern as sparse blob clone, but for trees.

- New flag: `sgit clone --with-history` for the old behaviour.
- New on-demand fetch path: `Vault__Sync.fetch_tree(tree_id)` already
  exists conceptually; we wire it into the history-touching commands.
- Backward-compat: yes, the format is unchanged; the lazy mode is a
  client-side decision.
- Migration: not needed.

Expected impact: 184s → likely under 10s for the case-study vault
(depends on HEAD tree depth).

### 7.2 Fix B — Tree pack format (server-side change, big win)

Server bundles many tree objects into a single binary "tree pack"
file, served as one HTTP. Same idea as Git's pack files. Two flavours:

- **Per-commit pack** — every commit gets a pack with all trees needed
  to reconstruct that commit. Clone HEAD = download HEAD's pack only.
- **Range pack** — server walks a commit range and produces one pack.
  Clone full = one pack containing all needed trees.

- Backward-compat: incremental — old vaults serve per-tree as today;
  new packs ship alongside.
- Migration command: needed for existing vaults to produce packs (the
  rekey precedent applies — we already have a "rebuild a vault"
  command surface).

Expected impact: 184s → seconds (single HTTP).

### 7.3 Fix C — Parallel decryption (CPU-bound portion)

Python `multiprocessing` (or `concurrent.futures.ProcessPoolExecutor`)
to decrypt trees in parallel. Useful only if H2 dominates (per-tree
decrypt overhead, not HTTP latency).

- Backward-compat: yes.
- Migration: not needed.

Expected impact: depends on H2 magnitude. If decrypt is 30s of the
184s, parallel-by-4 saves ~22s.

### 7.4 Fix D — Clone manifest (server-computed object list)

Server precomputes "for commit X, here are all object-ids you need" —
returned in one HTTP. Client then downloads (in parallel batches,
deduplicated locally).

- Backward-compat: incremental; old vaults compute the manifest on
  read; cached server-side after first request.
- Migration: not needed.

Expected impact: smaller than A or B but easier to ship; reduces
round-trips.

### 7.5 Recommended sequence for Phase 3

If Phase 2 confirms H5 dominates:
1. **Fix A first** — lazy historical tree fetch. Pure client-side. Big
   win, no protocol change.
2. **Fix B second** — tree pack format for the residual cases (full
   clone, large HEAD trees). Migration command produces packs for
   existing vaults.
3. **Fix C and D** are queued; revisit after A + B land.

If Phase 2 reveals H1/H2/H4 dominate instead, we re-rank.

## 8. Phase 4 — Extend to push / pull / fetch

After clone is fast, the same pattern applies:

- **Push**: today walks the local tree to find new objects; the same
  per-tree overhead may apply. The HMAC-IV change should already help
  here (deterministic IDs → server can say "I have this", we skip
  upload), but we haven't measured.
- **Pull**: similar to clone in shape; debrief 06 reported "146s → 3s"
  via BFS-batch tree download. Worth re-measuring with the new tools.
- **Fetch**: largely the same path as pull's tree walk.

Phase 0's `sgit dev profile <command>` should generalise across
clone / push / pull / fetch from the start, so we measure all four
the same way.

## 10. Boundary calls (route via Dinis)

Some items in this strategy cross the Villager / Explorer boundary
(Villager hardens what exists; Explorer designs new architecture).
Specifically:

- **Fix A** (lazy historical tree fetch as default) changes default
  clone behaviour. That's a product / Explorer call. Dinis decides.
- **Fix B** (tree pack format) is a new format on the server. Pure
  Explorer territory. The strategy here flags the design space; the
  detailed design happens in an Explorer session.
- **Sub-command consolidation** changes the CLI's user-facing surface.
  Likely Explorer + Designer; but with backward-compat aliases the
  hardening flavour is preserved.

The Villager team can do all of Phase 0 (instrumentation), much of
Phase 1 (the renames + aliases), and Phase 2 (diagnosis) without
crossing into Explorer territory. Phases 3 and 4 need explicit
Explorer sign-off on the architecture before Dev executes.
