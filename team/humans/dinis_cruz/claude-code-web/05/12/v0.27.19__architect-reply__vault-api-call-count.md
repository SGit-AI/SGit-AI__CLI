## TL;DR

**The protocol is already shaped to do what you want — the browser client isn't
using it the same way the CLI does.** Your two top-ranked smells
(`commit + push = 2 ref writes` and `clone-ref 404 on open`) **don't exist in the
CLI today.** The fix for both is client-side: adopt the existing
`execute_batch()` write path, and treat the clone ref as derived local state.
No new endpoints needed for the highest-impact items.

---

## a) Are you doing the right thing?

**The analysis is sharp and the right questions are being asked**, but the map
paints a worse picture than reality at the protocol level. The protocol already
supports atomic multi-op writes and the CLI exercises that path for every push.
What the browser is observing as "5 HTTP writes per single-file save" is, on
the CLI, **1 HTTP write** that bundles the blob + tree + commit +
named-ref-advance.

**Correctness of the map (from the browser POV):** mostly accurate. The browser
client genuinely is doing those round-trips — not disputed. What is disputed is
that any of it requires *protocol* changes.

**Key mismatches with the CLI:**

| Op | Web map | CLI reality |
|---|---|---|
| Single-file save | 5 writes (1 blob + 1 tree + 1 commit + 2 refs) | **1 write** — `Vault__Batch.execute_batch()` packs all four into one HTTP call (`sgit_ai/core/actions/push/Vault__Sync__Push.py:263`) |
| Cold open clone-ref read | 404 round-trip | **0 round-trips** — clone ref is created locally at clone time (`sgit_ai/workflow/clone/Step__Clone__Create_Clone_Branch.py:59`); never read from server |
| Status | Multiple ref reads | **0 network calls** — `Vault__Sync__Status` reads only local files |
| Divergence ancestor walk | Client-side DAG walk | Same — `Vault__Fetch.find_lca` walks locally. No server merge-base endpoint exists or is planned in the near term. |
| Push batching | Not modelled | 50-op chunks via `Vault__API.execute_batch`, presigned S3 fallback on 502/503 |
| Ref pointer caching | Implied (Q3) | None — CLI reads are local-file, so caching is unnecessary on CLI |

## b) Quick wins / bugs (browser-side only, no protocol change)

These four browser-client changes deliver the bulk of the wins your brief is
chasing. None require server work.

### 1. Stop writing the named ref in `_vault.push()` as a separate HTTP call
**(your Issue #1 — biggest win)**

Include the ref-advance op in the same batch as the commit/tree/blob writes via
`vaultBatch()`. The CLI's batch endpoint already accepts ref-write ops mixed
with object-write ops — see `Vault__Batch.build_push_operations`.

**Expected reduction:** 5 writes → **1 write per save**.

### 2. Stop reading the clone ref on open
**(your Issue #2)**

Create it locally at clone-time (matching the CLI). The "no clone ref means use
named HEAD" semantics is exactly what the CLI does — the clone ref is a
**client-owned** pointer, not a server-owned one. Treating it as server-side is
the root cause of the 404 read.

### 3. Skip the branch-index read for single-named-branch vaults
**(your Issue #6)**

The CLI also reads this unconditionally today, so this is a shared opportunity.
A flag in the vault metadata (e.g. `has_multi_branch: bool`) injected at
`sg.app.*` boot-time would let the browser skip the read. Easy and worth doing
on both clients.

### 4. Client-side ref-pointer TTL cache
**(your Issue #3 / Q3)**

A 2–5s TTL is reasonable for the browser. The CLI doesn't need this because its
ref reads are local-file. **Do NOT use `Cache-Control: max-age=2` on the
response** — refs are write-after-read sensitive, and a stale cache from CDN or
proxy is harder to debug than a client-owned TTL. Keep it in the JS layer keyed
by `(vault_id, ref_id)` and invalidate on any ref-write op from the same client.

## c) Bigger architecture / resilience changes

### 5. `execute_batch()` is your protocol-level fusion endpoint — it already exists

You don't need a new `POST /bare/refs/advance`. The atomicity guarantee you
want (commit + ref advance succeed-or-fail together) comes from putting both ops
in the same batch call. See `sgit_ai/network/api/Vault__API.py:65-96`
(`MAX_BATCH_OPS = 50`, presigned S3 fallback on 502/503). Mirror this on the
browser side and your Issue #1 evaporates.

### 6. Server-side merge-base (your Q5) — defer

Not planned, and I'd push back on adding it. Deep DAG walks are pathological
today only when the history is genuinely deep, and the CLI's batch-of-50 BFS
keeps wall-clock low even for vaults with thousands of commits. A server-side
LCA endpoint adds protocol surface and trust assumptions (the server now knows
commit graph shape) without a clear win.

**Better lever:** cache the LCA result client-side keyed by
`(local_head, remote_head)` — it's deterministic. Revisit if vaults regularly
hit pathological depths in real usage.

### 7. Subtree expansion batching (your Issue #7)

The CLI's `sparse_fetch` batches blob reads but expands trees one-at-a-time
too. Worth adding a `read_subtree_with_children(tree_id, depth=1)` to the
protocol; **both clients would benefit.** This is the one place I'd consider a
real new endpoint.

Migration shape: additive (new endpoint, old path stays). Today's vaults keep
working unconditionally.

### 8. Resilience win for both clients

The CLI has presigned-URL fallback on 502/503 for large blobs
(`Vault__API.py:84-95`). The browser should mirror this — your `vaultReadLarge`
path is the right shape; just make sure the 502/503 retry-with-presigned exists
on **writes too**, not only reads.

What "good" looks like, measured against the CLI today:

| Operation | Target | CLI today | Browser today |
|---|---|---|---|
| Vault open (cold, 1-commit vault) | **2** (branch_meta batch + tree+blob batch) | 4–5 (room to merge index + branch_meta) | 5+ (with 404 on clone-ref) |
| Pull when up-to-date | **1** (named ref read) | 1 ✓ | 1 |
| Pull when behind | **2** (ref read + 1 batch for missing objects) | 2–3 | 3+ (unconditional clone-ref write) |
| Status | **0** | 0 ✓ (local-only) | 2 (both ref reads) |

| Subtree expansion | **1** per depth-N tree | 1 per single subtree (room to grow) | Same |

## Answers to your 6 questions

### Q1. Fuse commit + push into a single operation?
**Already done at the protocol level — `execute_batch()`.** Browser needs to
adopt it. No new endpoint required.

### Q2. Faster first-open?
**Treat clone ref as locally-derived** (mirror the CLI). Zero protocol change.

### Q3. Ref TTL cache?
**Yes, 2–5s, client-owned**, not HTTP `Cache-Control`. Invalidate on any
ref-write op from the same client.

### Q4. Optimal batch size?
**50 is correct** for the current S3 backend. Throughput plateaus around 30–60
based on the CLI's measurements. Don't change.

### Q5. Server merge-base endpoint?
**Defer.** Cache LCA client-side instead. Revisit only if real-world deep-history
cases emerge.

### Q6. Auto-push semantics — does the CLI want unpushed state?
**Yes**, but only for the human-driven `sgit commit && sgit push` workflow where
users batch related work. For the iframe / `sg.vfs.write()` model, auto-push is
the right default. **The protocol does not care** — it's a client-UX choice.
Your auto-push behaviour holds at the protocol level.

## Migration shape

None of the highest-impact changes need protocol versioning — they're
browser-client adoptions of capabilities the protocol already has:

- **Items 1–4 (quick wins):** pure browser-client work, no server changes, no
  vault format changes, no backwards-compatibility concerns.
- **Item 7 (subtree-with-children):** new endpoint, additive. Old path stays.
  Old vaults keep working unconditionally.
- **Item 8 (presigned fallback on writes):** transparent — clients try
  `execute_batch` first, fall back to presigned on 502/503, exactly as the read
  path does today.
