## Deferred: Brief 06 Workstream C (new features)

| # | Feature | Effort | Notes |
|---|---|---|---|
| C-1 | `sgit tag` — lightweight content tags | ~2 days | **Anticipate signed tags from day one** — `Schema__Tag` should leave room for a signature field even if v1 doesn't populate it. PKI is on the v0.13+ roadmap. |
| C-2 | `sgit diff --remote` — diff local HEAD vs named branch without pulling | ~1 day | Use existing `sparse_ls` + named-branch metadata fetch. Don't re-implement the "fetch metadata, don't materialize" dance. |
| C-3 | `sgit log --json` — machine-readable commit history | ~½ day | **Schema decision is the gating question.** Recommend `Schema__Log_Entry` (Type_Safe) with: `commit_id`, `parents[]`, `timestamp`, `message`, `tree_id`, optional `signature`. The whole `--json` family (`ls --json`, `cat --json`, future `show --json`) should derive from a coherent schema family. |
| C-4 | `sgit export` — dump HEAD tree as plain files | ~2 days | Three open design questions: (a) format `dir | tar.gz | zip`, default tar.gz matching git? (b) preserve mtimes / permissions / symlinks? (c) decryption-boundary confirmation prompt or `--I-know-what-im-doing` flag? |

**Rationale for deferral:** C-features are net-new functionality. The
v0.12.x perf sprint is the higher-value lever. Once perf is solid,
the C-features benefit from a faster, more modular codebase.

**Cross-feature schema consideration:** C-3 sets the precedent for
`--json` output across the CLI surface. Decide `Schema__Log_Entry`
**before** any further `--json` flag ships on existing commands —
otherwise we ship two formats and have to converge them later.
