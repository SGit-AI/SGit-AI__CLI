## 1. Executive Summary

The codebase is **architecturally sound** with a well-layered structure, correct dependency direction, and strong type safety. The v0.16.3 implementation (self-describing IDs, sub-tree model, encrypted metadata) is well-integrated across all layers.

This review identifies **12 findings** across six categories: Type_Safe violations, dead code, algorithmic gaps, security posture, code quality, and test coverage. Of these, 2 are blocking, 3 are high priority, and 7 are medium/low.

**Overall health:** 8/10 — strong fundamentals, clear gaps to close.

## 2. Architecture Overview

### 2.1 Package Structure (Dependency Direction)

```
                    ┌──────────────┐
                    │     cli/     │  CLI entry points
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │    sync/     │  Orchestration (Vault__Sync, Merge, Batch, etc.)
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐

       ┌──────▼──────┐ ┌──▼────────┐ ┌─▼──────────┐
       │  objects/    │ │   api/    │ │   crypto/   │
       │ (store/refs) │ │ (network) │ │ (AES/PKI)  │

              │
       ┌──────▼──────┐
```

**Assessment:** Dependency direction is correct — inward only. `schemas/` and `safe_types/` have zero imports from upper layers. `sync/` orchestrates but doesn't leak domain logic upward. No circular dependencies detected.

### 2.3 Key Strengths

1. **Zero plaintext leakage to schemas.** `Schema__Object_Tree_Entry` has only `_enc` fields. `Schema__Object_Commit` has `message_enc`. Plaintext exists only in-memory as flat dicts from `Vault__Sub_Tree.flatten()`.

2. **Self-describing IDs everywhere.** All object references use the 4-segment convention (`obj-cas-imm-{hash}`, `ref-pid-muw-{hmac}`, `idx-pid-muw-{hmac}`, `key-rnd-imm-{random}`). Validators enforce format at the type boundary.

3. **Sub-tree model well-integrated.** `Vault__Sub_Tree` is used by `init()`, `commit()`, `status()`, `pull()`, `push()`, `clone()`, `Vault__Bare`, and `Vault__Inspector`. No flat-tree code paths remain in the active flow.

4. **Batch API is production-ready.** `Vault__Batch` builds push operations with CAS (`write-if-match` on refs), recursive tree collection, and graceful fallback to individual operations.

5. **Branch model is complete.** Named branches, clone branches, branch index, EC P-256 key pairs, commit signing, merge provenance on pull side — all working.

## 3. Findings

### Finding 1: `Vault__Components` Uses Raw Primitives (BLOCKING)

**File:** `sg_send_cli/sync/Vault__Components.py`
**Severity:** BLOCKING — violates CLAUDE.md Rule #1

```python
class Vault__Components(Type_Safe):
    vault_key              : str  = ''    # should be Safe_Str__Vault_Key
    vault_id               : str  = ''    # should be Safe_Str__Vault_Id
    read_key               : bytes = b''  # acceptable (crypto material)
    write_key              : str  = ''    # should be Safe_Str__Write_Key
    ref_file_id            : str  = ''    # should be Safe_Str__Ref_Id
    branch_index_file_id   : str  = ''    # should be Safe_Str__Index_Id
    sg_dir                 : str  = ''    # should be Safe_Str__Vault_Path or similar
```

Six of seven string fields use raw `str`. This is the most prominent Type_Safe violation in the codebase. `bytes` for `read_key` is acceptable since there's no `Safe_Bytes` primitive.

**Fix:** Replace each field with its corresponding Safe_Str type. Fields like `vault_key` and `sg_dir` may need `allow_empty = True` on their Safe_Str types or new Safe_Str subtypes.

### Finding 2: Raw Primitives in Schema Classes (HIGH)

**Severity:** HIGH — violates CLAUDE.md Rule #1

Three schema classes use raw primitives:

1. **`Schema__Object_Commit.attestations`** (`sg_send_cli/schemas/Schema__Object_Commit.py:21`):
   ```python
   attestations : list[str]   # reserved
   ```
   Should be `list[Safe_Str__Signature]` or a dedicated `Safe_Str__Attestation` type.

2. **`Schema__Change_Pack.payload`** (`sg_send_cli/schemas/Schema__Change_Pack.py:17`):
   ```python
   payload : list[str]   # list of file IDs
   ```
   Should be `list[Safe_Str__File_Id]`.

3. **`Schema__Vault_Policy`** (`sg_send_cli/schemas/Schema__Vault_Policy.py:9-10`):
   ```python
   require_author_signature : bool = False
   require_attestation      : bool = False
   ```
   Raw `bool` fields on a Type_Safe class. Should use a Safe_Bool type or be modelled as an enum.

### Finding 3: Vault ID Regex Rejects Valid IDs (BLOCKING)

**File:** `sg_send_cli/safe_types/Safe_Str__Vault_Id.py:5`
**Severity:** BLOCKING — prevents real vault operations

```python
VAULT_ID__REGEX = re.compile(r'^[0-9a-f]{8}$')  # hex only
```

But `Vault__Sync.generate_vault_key()` at line 42 generates vault IDs from `string.ascii_lowercase + string.digits` — which includes `g-z`. A generated vault ID like `12cpxeq9` would be rejected by the Safe_Str validator.

**Fix:** Change regex to `^[a-z0-9]{8}$`.

**Note:** Already identified in the conformance assessment. Repeated here because it's a blocking issue that affects every new vault creation where the random ID contains non-hex chars (probability: ~73% per 8-char ID).

### Finding 4: Dead Flat-Tree Code Will Crash If Reached (HIGH)

**File:** `sg_send_cli/sync/Vault__Sync.py`
**Severity:** HIGH — crash paths

Three dead code areas reference `entry.path` and `entry.name` — fields that no longer exist on `Schema__Object_Tree_Entry`:

1. **`_checkout_tree()`** — flat-tree fallback path (if it still exists; code around line 863-891 in previous versions)
2. **`_entry_path()`** — references `entry.path` and `entry.name`
3. **`_remove_deleted_files()`** — flat-tree branch accessing `_entry_path()`

**Current state:** These methods are NOT called from any active code path. The codebase uses `_checkout_flat_map()` and `_remove_deleted_flat()` instead (lines 894-916). However, the dead methods remain importable and would crash if called.

**Fix:** Delete `_checkout_tree()`, `_entry_path()`, and the flat-tree branch of `_remove_deleted_files()`. Only `_checkout_flat_map()` and `_remove_deleted_flat()` should remain.

**Note:** Reviewing the current code, these methods may have already been removed in the latest dev merge. Verify and delete if still present.

### Finding 5: Clone Uses `list_files()` Instead of Graph-Walking (HIGH)

**File:** `sg_send_cli/sync/Vault__Sync.py:788`
**Severity:** HIGH — algorithmic gap

```python
remote_files = self.api.list_files(vault_id, 'bare/')
for i, file_id in enumerate(remote_files, 1):
    data = self.api.read(vault_id, file_id)
```

Downloads ALL files including historical commits, not just HEAD-reachable ones. Makes N+1 API requests instead of the spec's 4+depth. Requires the `list` endpoint which should eventually be authenticated.

**Fix:** Rewrite clone to: derive ref → read HEAD → walk commit → tree → sub-trees → blobs → read branch index → fetch keys. Use `batch_read()` for efficiency.

### Finding 6: Push Does Not Create Named Branch Merge Commit (MEDIUM)

**File:** `sg_send_cli/sync/Vault__Sync.py:575`
**Severity:** MEDIUM — missing provenance

```python
ref_manager.write_ref(named_ref_id, clone_commit_id, read_key)
```

The push fast-forwards the named ref directly to the clone commit. No merge commit is created on the named branch, so:
- Named branch history is indistinguishable from clone branch history
- No merge provenance boundary
- Named branch has no independent signature chain

The pull side correctly creates merge commits (line 390-396). The gap is push-side only.

**Fix:** After computing delta, create a merge commit `parents=[named_commit_id, clone_commit_id]` signed with the named branch key, and point the ref at that merge commit.

### Finding 7: `_fetch_missing_objects()` Doesn't Recurse Into Deep Sub-Trees (MEDIUM)

**File:** `sg_send_cli/sync/Vault__Sync.py:959-1043`
**Severity:** MEDIUM — correctness at depth

The fetch helper walks one level of tree entries and downloads blobs + immediate sub-trees, but does NOT recursively descend into sub-trees. The comment at line 1030-1033 acknowledges this:

```python
# Note: deeper sub-trees will be discovered when their
# parent sub-tree is loaded in a future iteration. For
# full recursive fetch, the sub_tree.flatten() approach
# in clone is preferred.
```

For directories nested >1 level deep, the pull flow may miss sub-tree objects until a subsequent pull. This is mitigated by the `Vault__Sub_Tree.checkout()` which will attempt to load them, but those objects might not have been downloaded yet.

**Fix:** Use `Vault__Sub_Tree.flatten()` to discover all reachable objects, or recursively queue sub-tree IDs for download.

### Finding 11: No `local/tracking.json` for Fast LCA (LOW)

**Severity:** LOW for correctness, MEDIUM for performance at scale

LCA computation walks the full commit chain every pull. For vaults with long histories, this will degrade. A tracking file recording last-known ref positions would enable fast-path LCA.

## 5. Security Posture

### Strengths

1. **Zero-knowledge boundary is enforced by architecture.** The server never sees plaintext filenames, sizes, content hashes, or commit messages. All metadata is AES-GCM encrypted before storage.

2. **Cryptographic primitives are correctly used.** PBKDF2 at 600k iterations, HKDF-SHA256 for key derivation, AES-256-GCM with 12-byte IV, ECDSA P-256 for commit signing.

3. **HMAC-derived entry points prevent enumeration.** Ref and index file IDs are deterministic from vault key, not guessable.

4. **Commit signatures provide integrity chain.** Every commit is signed with the branch's EC private key.

5. **CAS on push prevents races.** `write-if-match` on ref updates ensures no lost updates on concurrent pushes.

### Concerns

1. **`read()` endpoint has no authentication.** `Vault__API.read()` sends no access token or write key. Anyone who knows the vault ID and file path can read encrypted blobs. This is acceptable if the encryption is the access control, but should be documented.

2. **API error messages include partial credentials.** `_api_error()` at line 127 logs first 8 chars of tokens/keys. Should be `***` or omitted entirely.

3. **`list_files()` exposes vault structure.** Even without read keys, an attacker can enumerate file IDs (which reveal object types via the self-describing prefix). This endpoint should be authenticated or removed (per spec).

### Import Hygiene

Two inline imports in `Vault__Batch`:
```python
# Line 44-45
from sg_send_cli.objects.Vault__Commit import Vault__Commit
from sg_send_cli.crypto.PKI__Crypto   import PKI__Crypto
```

These exist to avoid circular imports. Acceptable but should be documented with a comment explaining why.

### Duplicate Code

`Vault__Sub_Tree.build()` and `build_from_flat()` share ~40 lines of identical directory-structure logic (lines 31-47 and 116-132). Could be extracted to a shared `_partition_paths()` helper.

## 9. Conclusion

The codebase is well-designed with correct layering, strong type safety (aside from `Vault__Components`), and a clean zero-knowledge boundary. The v0.16.3 implementation is thoroughly integrated.

The two blocking items (Vault ID regex and Vault__Components raw types) are quick fixes. The most impactful work remaining is the graph-walking clone (F5) and push merge commit (F6), which complete the architectural vision from the v0.13.30 branch model spec.

Test coverage is strong at the schema and crypto layers (100%) and adequate at the sync layer (83%), with the main gaps in CLI and API layers where integration tests provide coverage.
