## 1. Scope

The v0.10.30 sprint added 4 new CLI commands (`probe`, `delete-on-remote`, `rekey`,
`write`), reshaped pull, made push resumable, introduced sparse clone, and changed
a foundational crypto invariant (HMAC-derived IVs for tree metadata). The Architect
angle answers: did Sonnet's seven Explorer-style additions stay inside the
Explorer-defined component boundaries, did they preserve the vault format and CLI
contract, and did they introduce structural duplication or pattern drift? This is a
review, not a redesign — findings either confirm "boundary respected" or flag
"sent back to Explorer". Crypto correctness is shared with AppSec; CLI contract is
shared with Designer/QA; we own the boundary and pattern verdicts.

## 2. Focus Areas

### A. HMAC-IV crypto change (debrief 07)
- **Question:** Does deterministic-IV-for-tree-metadata preserve `Vault__Crypto`'s
  encrypt/decrypt contract, and does any caller still assume non-determinism?
- **Targets:** `sgit_ai/crypto/Vault__Crypto.py`, `sgit_ai/sync/Vault__Sub_Tree.py`,
  call-sites of `encrypt_metadata` (grep only).
- **Evidence:** signature diff of `encrypt_metadata`, list of callers, classification
  per caller (tree-metadata vs blob vs commit). Cross-link to AppSec on IV-reuse
  threat model.

### B. New commands vs Explorer-defined CLI boundaries
- **Question:** Do `probe`, `delete-on-remote`, `rekey`, `write` live in their proper
  `CLI__*` class, with logic in `Vault__Sync` (not in `cli/__init__.py`)?
- **Targets:** `sgit_ai/cli/__init__.py`, `CLI__Vault.py`, `CLI__Main.py`,
  `Vault__Sync.py`. Cross-check rule 7 of `CLAUDE.md`.
- **Evidence:** import/grep for each new command name; line count per CLI handler;
  flag any command whose body exceeds a thin dispatch.

### C. Vault format / CLI contract drift
- **Question:** Did the sprint silently change the on-disk vault layout, the wire
  envelope, or the textual output of any pre-existing command?
- **Targets:** `.sg_vault/local/clone_mode.json` (new), `push_state.json` (new),
  `Schema__Object_*`, any rendered string from `cat`, `ls`, `log`.
- **Evidence:** grep new schema fields, list new files written under `.sg_vault/`,
  diff CLI rendering for `cat`/`ls`/`log` between Apr 19 and May 1.

### D. Duplication & boundary violations between new and existing components
- **Question:** Does `Vault__Sync.write()` re-implement encrypt/store/commit logic
  already in `commit()` / `push()`? Does sparse-clone duplicate `_clone_*` helpers?
  Does pull's BFS duplicate clone's BFS instead of sharing it?
- **Targets:** `Vault__Sync.py` (22 changes — hottest file), `Vault__Batch.py` (3),
  `Vault__Sub_Tree.py` (2), `Vault__Diff.py` (3).
- **Evidence:** function-name similarity scan, side-by-side outline of
  `clone`/`pull`/`write`/`rekey` pipelines, list of helpers that are now duplicated
  vs extracted.

### E. Type_Safe & naming hygiene on additions
- **Question:** Do new schemas/safe-types follow the `Schema__*` / `Safe_Str__*`
  conventions, no raw primitives, immutable defaults?
- **Targets:** any new file under `sgit_ai/schemas/`, `sgit_ai/safe_types/`,
  new fields on `Schema__Object_Tree_Entry`, `push_state` structure.
- **Evidence:** Review-checklist sweep from `architect__ROLE.md`. Round-trip
  invariant on any new schema.

### F. Resumable push state machine
- **Question:** Is `push_state.json` a forward-compatible schema, and is its
  cleanup path safe across crash/abort/interrupt?
- **Targets:** `Vault__Sync.py` push code, any new `Schema__Push_State`.
- **Evidence:** state transitions, key `{vault_id}:{commit_id}` collision review,
  fail-modes when checkpoint diverges from server.

### G. Sparse clone mode & mode-promotion path
- **Question:** Is `clone_mode.json` a one-way switch, or can a sparse clone be
  promoted to full? What does `pull` do in sparse mode? Is the `✓`/`·` indicator a
  contract or an implementation detail?
- **Targets:** `Vault__Sync.clone()`, `_clone_with_keys`, `_clone_download_blobs`,
  `Vault__Inspector`.
- **Evidence:** call graph, mode-read sites.

### H. Pull BFS pattern reuse and progress-bar contract
- **Question:** Was clone's BFS extracted to a shared helper or copied? Is the
  progress-bar denominator now a contract that downstream tooling depends on?
- **Targets:** `Vault__Sync._fetch_missing_objects`, `Vault__Sync._clone_*`,
  `CLI__Progress.py`.
- **Evidence:** structural diff between clone's and pull's BFS; commit `0c29c72`.

## 3. Approach

**Top-down.** The seven debriefs already describe intent at the right level —
re-reading source line-by-line is not the best use of time. For each focus area:

1. State the architectural claim from the debrief.
2. Spot-check with `grep`/file-outline only (no full-file reads beyond what is
   strictly needed to confirm/deny the claim).
3. Record verdict: `BOUNDARY OK` / `BOUNDARY DRIFT` / `SEND-BACK-TO-EXPLORER`.

**Coordination points:**
- **Dev (sibling):** They will own duplication-fix proposals; we only flag.
  Share findings on areas D and H before they draft refactor tickets.
- **AppSec (sibling):** Area A (HMAC-IV) is jointly owned — they cover the
  cryptanalytic risk, we cover whether the change leaked across the
  `Vault__Crypto` / `Vault__Sub_Tree` boundary.
- **Sherpa:** Anything tagged `SEND-BACK-TO-EXPLORER` becomes a decision point.

## 5. Risks / Unknowns

- HMAC-IV change is foundational; old vaults on disk may produce different tree
  IDs than new clients — backward-compat for existing servers is unclear from
  debrief 07 alone.
- `Vault__Sync.py` received 22 commits in 2 weeks; structural drift is plausible
  even if each individual commit looked clean.
- `sgit write` may have created a second "commit pipeline" parallel to the
  scan-based one — duplication risk is high.
- `push_state.json` and `clone_mode.json` add new on-disk artifacts inside
  `.sg_vault/local/`; their place in the vault format spec needs confirmation.
- Sparse clone changes the semantics of "vault is fully present locally" — pull,
  status, and reset behaviour in sparse mode is not described in the debriefs.
- `--json` flags on `cat`/`ls` are now part of the public CLI contract; not yet
  documented as such in any briefing pack.
- Progress-bar denominator change (`1069` → real count) may itself be a CLI
  contract change for any wrapper script that parses the output.
- Test count (2104) suggests broad coverage but says nothing about boundary
  tests; QA will need to confirm.

## 6. Out of Scope

Per Villager rules, this work explicitly does **not** include:

- Redesigning crypto, sync, or CLI components.
- Adding new commands, schemas, or safe types.
- Changing the vault format or CLI contract — only verifying what changed.
- Producing refactor patches (Dev sibling owns Phase 3 patches if needed).
- Threat modelling or cryptanalysis of HMAC-IV (AppSec owns).
- Test-coverage measurement (QA owns).
