## 1. Scope

A code-quality review of **what was shipped this sprint**, applied through the
Villager Dev lens (Type_Safe compliance, no-mocks discipline, naming, duplication,
error handling, edge-case test quality). The review is **read-only** and produces
findings only — no source or test edits, no commits.

The plan answers, for each new capability cluster:

- Are new schemas / payload classes free of raw `str` / `int` / `dict`?
- Do new CLI commands obey the `CLI__*` class pattern and the
  `cli/__init__.py` purity rule?
- Were any `unittest.mock` / `patch` / `MagicMock` usages **introduced** in
  this sprint (vs. pre-existing baseline)?
- Are `__init__.py` files absent from the `tests/` tree?
- Is duplication appearing across the new commands (probe vs clone token-handling,
  rekey vs init/commit, write vs commit, fetch vs clone blob download)?
- Are error paths consistent (failure of rekey mid-way, push checkpoint orphan,
  CAS conflict messaging, missing branch-index on probe)?
- Does test coverage on new code meet the Villager standard
  (real objects, no mocks, edge cases, round-trip invariants)?

The previous Villager Dev baseline (v0.5.11) reported 18 mock violations, 3 raw-primitive
files, and ~94% Type_Safe compliance. This review establishes a **delta** against that
baseline for the v0.10.30 sprint code only.

## 3. Approach

1. **Inventory pass (read-only).** Use `git log --since` filtered to v0.10.30 commits to enumerate every changed source file and every new test file. Build a working set of ~15–20 files.
2. **Static scan pass.** Run targeted greps from `tests/` and the working set: `: str`, `: int`, `: dict` (Type_Safe field declarations), `unittest.mock`, `MagicMock`, `@patch`, `assert ` outside tests, bare `Exception`, `print(` outside `CLI__Progress`. Capture line:file evidence.
3. **Read pass per focus area.** For each focus area, open the smallest necessary slice, capture quotes (file + line range), and write a finding. Do not read whole modules unless duplication scan demands it.
4. **Test-quality sampling.** Open every new test file in full. Confirm each test class follows `Test_<Class>` naming, uses `setUpClass`/`setUp` for real fixtures, and asserts behaviour rather than internals.
5. **Coordination touchpoints:**
   - **Architect** owns module-split decisions (focus area #6) and any boundary changes implied by duplication findings (focus area #5). Villager Dev flags, does not propose splits.
   - **AppSec** owns the HMAC-IV cryptographic review (focus area #9). Villager Dev only checks that determinism / interop / round-trip tests exist.
   - **QA** (if engaged) owns coverage gaps. Villager Dev only flags missing edge cases for the seven new clusters.
6. **Findings are descriptive, not prescriptive.** Each finding states: what, where (file:line), why it matters (rule reference), severity (info/minor/major/blocker), suggested next-action owner (Villager Dev fix / Architect decision / Explorer rework / AppSec review).

## 5. Risks

- **R1 — Type_Safe violation that requires architecture clarification.** If a new schema uses `dict` because the field is a free-form payload (e.g., per-blob checkpoint), Villager Dev cannot unilaterally introduce a Safe_* type — escalate to Architect. Likelihood: medium.
- **R2 — Behaviour-change disguised as refactor.** A duplication finding (focus area #5) might tempt a future refactor that subtly alters semantics (e.g., probe vs clone token-derivation diverging on edge cases). Mark such findings *do-not-refactor-without-tests-first*. Likelihood: medium.
- **R3 — HMAC-IV backward-compat gap.** If no test asserts decrypt of pre-sprint random-IV ciphertext, this is a **major** finding and a blocker for any future test suite hardening. Likelihood: low–medium.
- **R4 — Mock regression.** Sprint debriefs do not mention test-suite hygiene; new test files may have pulled in patches for convenience under deadline pressure. Baseline was 18 violations; if the count grows, escalate. Likelihood: medium.
- **R5 — `Vault__Sync.py` at ~3000 LOC.** Past hard cap for ergonomic review; pure-read review still feasible but Architect should be put on notice. Likelihood: high (already true).
- **R6 — Out-of-scope drift.** Reviewer may notice opportunities to refactor, fix bugs, or add features. **Defer all such items to debrief; do not act.**

## 6. Out of Scope (per Villager Dev role)

- No feature additions, no new CLI commands, no new vault operations.
- No bug fixes that change behaviour. If found, file as Explorer hand-back with reproduction.
- No aesthetic-only refactors (rename for taste, reorder methods, etc.).
- No new dependencies.
- No edits to `team/humans/dinis_cruz/briefs/` (READ-ONLY).
- No changes to existing tests (even mock-removal) — that is a separate Phase 3 refactor cycle.
- No work on Architect / AppSec / QA / Designer / DevOps / Sherpa angles — those are parallel agents.
