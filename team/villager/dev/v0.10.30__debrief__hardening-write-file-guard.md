## What Was Done

### Guard Added (1 location)

| Location | Guard | Effect |
|---|---|---|
| `sgit_ai/sync/Vault__Sync.py:246` | `if not c.write_key: raise Vault__Read_Only_Error()` | Rejects all writes when `write_key` is absent — i.e. for any read-only clone |

The guard fires immediately after `_init_components()` returns, before any I/O (no blob written, no ref updated, no working-directory file touched).

### `_init_components` Hardened (fail-closed)

| Condition | Previous behaviour | New behaviour |
|---|---|---|
| `clone_mode.json` unparseable (invalid JSON) | Exception from `json.load` swallowed → fell through to full-mode, granting write access | Raises `Vault__Clone_Mode_Corrupt_Error` |
| `clone_mode.json` present, `mode=read-only`, missing `read_key` or `vault_id` | Attempted read-only init with `None` → undefined behaviour | Raises `Vault__Clone_Mode_Corrupt_Error` |
| `clone_mode.json` absent | Full-mode (vault_key loaded) | Unchanged |
| `clone_mode.json` present and valid | Read-only or full depending on `mode` field | Unchanged |

**Decision: fail-closed.** A corrupt `clone_mode.json` raises rather than defaulting to full-mode. The operator must fix or re-clone. This is the correct trade-off: a transient file-system error should never silently grant write access to a read-only clone.

### Exception Classes (1 file)

`sgit_ai/sync/Vault__Errors.py` — two typed exception classes with module-level message constants:

| Class | Constant | Raised when |
|---|---|---|
| `Vault__Read_Only_Error` | `MSG_WRITE_KEY_MISSING` | `write_file` called on a vault with no `write_key` |

Constants allow `assert 'read-only' in str(err)` style tests without duplicating the message text at test sites.

## What Was Learned

1. **The guard location matters.** Placing `if not c.write_key` immediately after `_init_components()` means every code path below is guaranteed to have a write key. The check is trivially auditable.

2. **Fail-closed is the only safe default for file-system anomalies.** Any `except: pass` on file parsing that falls back to permissive behaviour is a security bug in a vault context. Failing loudly forces the operator to investigate rather than silently continuing in a degraded state.

3. **Module-level constants make exception messages testable.** Testing `assert 'read-only' in str(err)` would be fragile if the string lived only at the raise site. Promoting it to a named constant (`MSG_WRITE_KEY_MISSING`) lets tests import and compare it directly, and makes future renames a one-line change.

4. **The working-directory check is not enough.** Mutation M7 showed that `test_written_file_appears_on_disk` checks only the working directory, not the encrypted blob in `bare/data/`. This debrief confirms that the guard closes the `clone_mode.json`-corruption attack path, but the encryption-skip attack path (writing plaintext into the object store) remains open until brief 20 adds a ciphertext assertion.
