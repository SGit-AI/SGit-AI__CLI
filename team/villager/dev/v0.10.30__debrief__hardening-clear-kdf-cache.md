## What Was Learned

1. **Module-level cache is the right level.** Moving the cache inside the class (as a class variable or instance variable) would not work correctly with `functools.lru_cache` — the cache would be per-instance (instance method caches key on `self`) rather than shared. The module-level free function remains the canonical pattern; `clear_kdf_cache()` on the class is a thin, correct wrapper.

2. **Three probe return paths, not one.** `probe_token` has three exit points: successful vault probe, successful share probe, and the not-found `RuntimeError`. All three required `clear_kdf_cache()` to ensure no path leaks key material. The not-found path (before the `raise`) is the easy one to miss.

3. **Residual risk — Python byte zeroing.** Python does not provide reliable byte-level memory zeroing. Calling `cache_clear()` removes the LRU references to passphrase/key bytes, but the bytes objects may remain in process memory until garbage collection. This is noted as a known residual risk in both the brief and the F03 finding; it is out of scope for Python-layer fixes.

4. **Long-running agent context gap.** The probe + delete + rekey wiring covers the agent surface added this sprint. However, if a future agent holds a `Vault__Sync` instance across multiple vault operations (e.g., write → cat → ls in sequence for different vaults), the KDF cache will accumulate derived keys for all vaults touched in the session. This warrants an Architect brief on multi-tenant scope isolation — the current fix is correct but not sufficient for a multi-vault agent host.

## What Was NOT Changed

- The KDF itself — PBKDF2-SHA256 with 600,000 iterations, `maxsize=256`. No behaviour change.
- `_pbkdf2_cached` signature or module location — unchanged.
- No new dependency introduced.
- `rekey_init` and `rekey_commit` do not call `clear_kdf_cache()` — they are the follow-on steps after `rekey_wipe` and need the derived keys for the new vault; clearing after wipe is the correct boundary.

---

## Escalation Note (Architect)

The multi-tenant agent host scenario described in "What Was Learned" item 4 is out of scope for this brief (per F03 noting it was "escalated to Architect"). If SGit-AI agent usage grows to include multi-vault sessions (e.g., a daemon serving multiple vault identities), the LRU cache will accumulate derived keys for all active vaults. The correct architectural fix is per-session or per-vault-identity cache isolation, which requires a design decision on the `Vault__Sync`/`Vault__Crypto` instantiation model.
