## 2. Elevator Pitch

sgit-ai gives developers git-style workflows — clone, commit, push, pull, branch, merge — for encrypted vaults where the server never sees plaintext. Built on AES-256-GCM with byte-for-byte Web Crypto interop, sgit powers the SG/Vault browser experience and provides a CLI that treats encryption as infrastructure, not an afterthought. It is the engine behind the SGraph-AI encrypted ecosystem.

## 3. Value Propositions by Audience

**Message:** Stop bolting encryption onto your file sync. sgit gives you `init`, `clone`, `commit`, `push`, `pull`, `branches`, and `merge` — all client-side encrypted with AES-256-GCM. Every object is content-addressable and encrypted before it leaves your machine. Install with `pip install sgit-ai`, run `sgit init my-vault`, and you are working.

**Key points:**

- `pip install sgit-ai` and go — Python >=3.11, Apache-2.0
- Content-addressable encrypted object store with integrity verification (`sgit fsck`)
- Built-in PKI: generate keys, sign files, verify signatures, hybrid encrypt — no GPG required
- Three-way merge with conflict detection across encrypted branches
- Inspect internals: `sgit log`, `sgit cat-object`, `sgit inspect-stats`

### Security Teams

**Headline:** Zero-knowledge vault sync with verifiable crypto guarantees.

**Message:** sgit implements a zero-knowledge architecture where the server stores only ciphertext. Key material never leaves the client. Encryption uses AES-256-GCM with per-file HKDF-SHA256 derived keys and PBKDF2-SHA256 (600k iterations) for vault key derivation. Every operation produces output that is byte-for-byte identical to the Web Crypto API given the same inputs — verifiable, auditable, interoperable.

**Key points:**
- Zero-knowledge server — stores ciphertext only, no key material, no plaintext metadata
- AES-256-GCM + HKDF-SHA256 per-file key derivation — no key reuse across objects
- PBKDF2-SHA256 with 600k iterations for vault key derivation
- Web Crypto API byte-for-byte interop — cross-verify CLI output in any browser
- Built-in PKI: RSA-4096 + ECDSA P-256 keygen, sign, verify, hybrid encrypt/decrypt
- Content-addressable storage with `fsck --repair` for integrity auditing
- Apache-2.0 licensed — full source available for review

### Enterprises

## 4. SG/Vault Case Study Outline

### Title: "SG/Vault — Encrypted Vault Management Powered by sgit"

### Problem
Teams need to store and share sensitive files — credentials, keys, contracts, health records — but existing solutions force a choice: security (GPG-encrypted archives, no collaboration) or convenience (cloud storage, no real encryption). There is no tool that gives both encrypted storage and git-style collaboration with branching and merging.

### Solution
SG/Vault (vault.sgraph.ai) is a browser-based encrypted vault manager built on sgit. It provides:
- A web UI for browsing, uploading, and downloading encrypted vault contents
- Full commit history with branch support
- The same AES-256-GCM encryption as the sgit CLI — byte-for-byte identical output
- Zero-knowledge architecture where the server never handles plaintext

### How sgit Powers SG/Vault
- **Encrypted object store:** Every file uploaded through SG/Vault is encrypted client-side using sgit's crypto primitives (AES-256-GCM, HKDF-SHA256)
- **Version control:** SG/Vault uses sgit's commit, branch, and merge operations to provide version history and parallel workstreams
- **Sync protocol:** SG/Vault communicates with the Transfer API using the same protocol as `sgit push` and `sgit pull`
- **Web Crypto interop:** Because sgit produces byte-for-byte identical ciphertext to the Web Crypto API, the browser and CLI can operate on the same encrypted objects without translation

### Architecture (text diagram)

```
Browser (vault.sgraph.ai)        CLI (sgit-ai)
         |                              |
    Web Crypto API              sgit_ai/crypto/
         |                              |
         +---------- same AES-256-GCM ciphertext ----------+
                            |
                   SGit-AI Transfer API
                    (ciphertext only)
                            |
                   Encrypted Object Store
```

### Key Outcome
A user can `sgit push` from the CLI and immediately see the updated vault in SG/Vault's browser interface — or upload through the browser and `sgit pull` to their local machine. One encrypted vault, two interfaces, zero plaintext on the server.

### Proof Point
SG/Vault is live at vault.sgraph.ai, backed by sgit-ai v0.8.17 on PyPI. The tool is available under Apache-2.0 and can be independently verified.

### Features Section

**Section headline:** What sgit does

**Feature 1: Encrypted Vault Sync**
Clone, commit, push, and pull encrypted vaults. Every object is AES-256-GCM encrypted with per-file HKDF-derived keys before upload. Content-addressable storage means duplicate content is encrypted once.

**Feature 2: Branching and Merge**
Create branches, work in parallel, and merge with three-way conflict detection — all on encrypted data. The same workflows you use in git, applied to encrypted vaults.

**Feature 3: Zero-Knowledge Architecture**
The server stores ciphertext only. Key material stays on your machine. PBKDF2-SHA256 with 600k iterations derives vault keys. No plaintext, no metadata leaks, no trust required.

**Feature 4: Web Crypto Interop**
sgit's crypto output is byte-for-byte identical to the Web Crypto API. The same vault is accessible from the CLI and from the browser (via SG/Vault). Verify CLI output in any browser's console.

**Feature 5: Built-in PKI**
Generate RSA-4096 + ECDSA P-256 key pairs, sign and verify files, and perform hybrid RSA-OAEP + AES-256-GCM encryption — without GPG or external tools.

**Feature 6: Vault Inspection**
Debug and audit vault internals. `sgit log` for commit history, `sgit fsck --repair` for integrity checks, `sgit cat-object` to decrypt and view individual objects, `sgit inspect-stats` for object store analytics.

## 7. Differentiator Messaging

### Positioning Table

| Capability | sgit-ai | git-crypt | age | SOPS |
|-----------|---------|-----------|-----|------|
| Encrypted vault sync (clone/push/pull) | Yes | No (encrypts files in git repo) | No (file encryption only) | No (encrypts values in config files) |
| Branching and merge on encrypted data | Yes (three-way merge) | Inherits from git | N/A | Inherits from git |
| Zero-knowledge server | Yes (server stores ciphertext only) | N/A (uses git remotes) | N/A (no server) | N/A (uses git/cloud remotes) |
| Web Crypto API interop | Yes (byte-for-byte) | No | No | No |
| Built-in PKI (keygen, sign, verify, encrypt) | Yes (RSA-4096 + ECDSA P-256) | No | No (encrypt only) | No |
| Content-addressable encrypted storage | Yes | No | No | No |
| Browser access to same vault | Yes (via SG/Vault) | No | No | No |
| Vault inspection tools | Yes (log, fsck, inspect, cat-object) | No | No | No |

### Narrative Differentiators

**vs git-crypt:**
git-crypt encrypts individual files inside a git repository. sgit is the repository — a complete encrypted vault with its own object store, commit history, branching, and merge. git-crypt depends on git; sgit replaces it for encrypted data. git-crypt has no server-side zero-knowledge guarantee because the git remote sees repo structure. sgit's server sees only opaque ciphertext blobs.

**vs age:**
age is an excellent file encryption tool. sgit uses similar crypto primitives but adds everything on top: version control, branching, merging, sync, content-addressable storage, and vault inspection. age encrypts a file; sgit manages an encrypted vault over time. They solve different problems at different layers.

**vs SOPS:**
SOPS encrypts values inside configuration files (YAML, JSON, ENV) so the structure remains readable for diffing and code review. sgit encrypts entire files and provides its own version control. SOPS is purpose-built for config secrets in git repos; sgit is purpose-built for encrypted vault management with sync. If you need to encrypt a `.env` file and keep it in git, use SOPS. If you need to manage an encrypted vault of arbitrary files with clone/push/pull and zero-knowledge sync, use sgit.

### One-Liner Differentiators (for comparison pages or FAQ)

- **git-crypt encrypts files in git. sgit is git for encrypted vaults.**
- **age encrypts a file. sgit manages a vault.**
- **SOPS encrypts config values. sgit encrypts everything and adds version control.**
- **Only sgit gives you the same encrypted vault in the terminal and the browser.**

## Messaging Decisions Log

| Decision | Rationale |
|----------|-----------|
| Lead with "Git workflows" not "encryption library" | Developers relate to git. The git mental model is sgit's strongest hook. Encryption is the differentiator, not the headline. |
| Avoid "military-grade" and similar marketing language | Developer audience. They know what AES-256-GCM is. Respect their intelligence. |
| Position SG/Vault as hero, not sgit features | The ecosystem story (CLI + browser on same vault) is more compelling than a feature list. SG/Vault proves sgit works at product scale. |
| Acknowledge alpha maturity honestly | Per ROLE.md: "alpha software with strong crypto foundations is a valid position." Do not overclaim stability. |
| Use `sgit` not `sgit-ai` in CLI examples | The shorthand is cleaner and what users will type. Mention `sgit-ai` as the install name. |
| Concrete commands over abstract descriptions | "sgit clone \<vault-key\>" communicates more than "easily sync your encrypted vaults." Show the tool working. |
