## Step 2 — Naive parallel run

`pytest tests/unit/ -n auto -q`:

- Workers chosen by `auto`: **4** (`created: 4/4 workers`).
- Wall clock: **76.65s** on first run.
- Pass / fail: **2,105 passed, 0 failed**.

**No isolation failures appeared.** Every test that passes serial also
passes under `-n auto`. The suite is parallel-safe out of the box on
this host. This is consistent with the v0.10.30 baseline finding that
all slow tests use per-method `tempfile.mkdtemp()` — temp paths are
already unique per test, so no cross-worker collision.

## Step 3 — Non-parallel-safe tests

**None identified.** Marker budget (≤ 5 tests / 2 modules) was not
needed.

The `no_parallel` marker is registered in `pyproject.toml` so that:

1. Future tests with global state (port binds, fixed paths, singleton
   resources) have a recognised opt-out without re-touching config.
2. The CI two-pass invocation already supports the marker, so a
   single-line tag in a future test will route correctly without further
   pipeline work.

## Step 4 — CI invocation

Updated `.github/actions/pytest__run-tests/action.yml`. The
`parallel: "true"` path now runs **two passes**:

1. `pytest -n auto -m "not no_parallel"` (parallel, fast).
2. `pytest -m no_parallel` (serial). Exit code 5 ("no tests collected")
   is treated as success — the suite has no marked tests today.

Coverage uses `coverage run --source=sgit_ai` for the parallel pass and
`--append` for the serial pass, then a single `coverage report` /
`coverage xml`. This keeps a unified coverage file regardless of which
pass picks up which test.

The `parallel: "false"` path (used by `tests/integration/` and
`tests/qa/`) is unchanged — those suites stay serial.

## Step 5 — Timing matrix

| Configuration | Workers | Wall clock | Notes |
|---|---:|---:|---|
| Serial (sandbox baseline this brief) | 1 | 244.80s | This sandbox; baseline doc reports ~124s on cleaner host. |
| Parallel `-n auto` | 4 | **76.65s** | First run; cleanest. |
| Parallel `-n auto` (re-run #1) | 4 | 92.28s | Sandbox under load. |
| Parallel `-n auto` (re-run #2) | 4 | 96.04s | Sandbox under load. |
| Parallel `-n 4 --dist=loadfile` | 4 | 85.91s | Per-file dist; slightly worse than default loadbalancing. |
| Parallel `-n 8` | 8 | 86.89s | Oversubscribed (only 4 vCPUs); slower than `-n 4`. |
| Combined CI (parallel + serial pass) | 4 + 1 | **76.65 + 2.93 ≈ 79.6s** | First run. Marker pass is collection-only today. |
| Combined CI (re-run) | 4 + 1 | 92.28 + 2.93 ≈ 95.2s | Under sandbox load. |

Observations:

- **`-n auto` is the right choice.** Default load-balancing dist beats
  `loadfile` here. `-n 8` is oversubscribed on a 4-vCPU host and runs
  slower — `auto` handles this correctly by capping at logical CPU
  count.
- **No cold-vs-warm gap.** Cache effects are negligible (suite is
  CPU-bound, not I/O-bound) so warm timings are representative.
- **Serial pass is ~3s.** Just collection; today no test is marked.
  Negligible overhead even if 1–2 tests get tagged later.
