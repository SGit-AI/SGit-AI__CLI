## 1. Scope

Verify the **zero-knowledge guarantee** across the seven v0.10.30 capability clusters
under the Villager assumption: *the SGit-AI server (and any party with on-disk access
to a client checkout) is hostile*. The deliverable is a focused set of evidence-backed
findings that answer:

- Does any new code path expose plaintext, key material, derivation inputs, file
  names, file sizes, or content hashes outside the encrypted layer?
- Does the deterministic HMAC IV change preserve AES-GCM security guarantees, or
  does it open new oracles / cross-vault leakage / nonce-reuse risk?
- Do the new lifecycle commands (`probe`, `delete-on-remote`, `rekey`) handle key
  material safely on disk, in memory, in process exit, and in error paths?
- Do the new agent-friendly commands (`write`, `cat --id/--json`, `ls --ids/--json`,
  sparse clone, on-demand `fetch`) widen the side-channel surface visible to the
  server (access patterns, sizes, timing)?
- Does any new UX (pull progress, clone progress, log diff stats, error messages)
  print sensitive data to stdout/stderr/debug logs?
- Are dependencies (`cryptography`, `osbot-utils`) pinned at versions free of known
  CVEs?

Out of phase: writing tests, modifying code, redesigning crypto. Those are Phase 3.

---

## 2. Adversarial Frame

We assume:

| Adversary | Capability |
|-----------|------------|
| **Compromised server** | Full read of all stored objects, full read of all API calls, can correlate object IDs across users / vaults / time, can serve crafted responses |
| **Disk-forensic adversary** | Reads `.sg_vault/` after process exit; reads swap; reads core dumps; reads OS-level deleted-but-not-overwritten blocks |
| **Shoulder-surfer / log scraper** | Reads stdout, stderr, debug logs, shell history, CI logs |
| **Shared-process adversary** | Reads other tenant memory in same Python process (think Lambda warm starts, multi-tenant agents) |

A finding counts only if it ties to one of these adversaries.

## 3. Focus Areas (prioritised)

### F01 — Deterministic HMAC IV (commit `4d53f79`) — HIGHEST PRIORITY
- **Question:** Does `iv = HMAC(read_key, plaintext)[:12]` preserve AES-GCM
  nonce-uniqueness, IND-CPA (within a key), and zero cross-vault leakage? Does the
  plaintext input space (filename, size string, content_hash, content_type, tree
  JSON) contain low-entropy values that could enable IV/ciphertext correlation
  attacks if the same `read_key` is ever reused (same passphrase + same vault_id)?
- **Files:** `sgit_ai/crypto/Vault__Crypto.py` (`encrypt_deterministic`,
  `encrypt_metadata_deterministic`), `sgit_ai/sync/Vault__Sub_Tree.py` (call sites
  at lines 81–84, 106, 156–159, 181, 245), tests in
  `tests/unit/sync/test_Vault__Sync__Simple_Token.py`.
- **Evidence type:** Spec read + Python REPL trace (no source edits) + cross-vault
  ID comparison test designed but not executed.
- **Adversarial mutation:** Replace `hmac.new(key, plaintext, sha256)` with
  `hashlib.sha256(plaintext)` (Option A from debrief) — should be caught by an
  IND-CPA-within-vault test. Replace HMAC key with constant — should be caught by
  cross-vault leakage test.
- **Specific concern:** read_key is used directly as HMAC key AND AES key. Verify
  this is not a domain-separation violation (re-using the same key for HMAC-SHA256
  and AES-256-GCM is generally safe but worth one paragraph of justification).

### F02 — Rekey key-material lifecycle
- **Question:** During `rekey_wipe → rekey_init → rekey_commit`: (a) is the old
  `vault_key` file overwritten or simply unlinked? (b) Are the old `read_key_bytes`
  / `write_key_bytes` / `passphrase` ever cleared from `_pbkdf2_cached` LRU? (c) Is
  there a window where both old and new vault_key files coexist on disk? (d) If
  rekey aborts between wipe and init, what state is the working copy left in? (e)
  Do old encrypted blobs remain on the server (the debrief says yes; verify the
  CLI tells the user)?
- **Files:** `sgit_ai/sync/Vault__Sync.py` lines 1759–1801, `sgit_ai/crypto/Vault__Crypto.py`
  `_pbkdf2_cached`, `Vault__Storage` (vault_key file path), CLI__Vault rekey commands.
- **Evidence type:** Static read + on-disk inspection of a temp vault rekey
  trace via existing tests.
- **Adversarial mutation:** Skip `rmtree(.sg_vault/)` — does any test catch
  leftover keys? Skip `commit` step — is partial state detected?

### F03 — `_pbkdf2_cached` LRU memory residency
- **Question:** `functools.lru_cache(maxsize=256)` retains the last 256
  `(passphrase_bytes, salt_bytes)` keys in process memory indefinitely. After
  `rekey`, the old passphrase is still cached. After `probe_token` on an unknown
  token, the candidate passphrase remains. In a long-running agent process this
  is a key-material residency violation.
- **Files:** `sgit_ai/crypto/Vault__Crypto.py` line 27.
- **Evidence type:** Code reading; reference Python `gc.get_referents` behaviour.
- **Adversarial mutation:** None — this is a static finding to document.
- **Severity:** Medium (depends on process lifecycle; high for agent backends).

### F04 — Probe leakage
- **Question:** Does `probe_token` (a) leak vault_id existence to a passive
  observer beyond what an unauthenticated GET already does? (b) Print the
  derived vault_id in the "not found" error path (line 1836: `derived vault_id={vault_id}`)?
  This is a *hash of the simple token* — confirm whether printing it to stderr
  is acceptable. (c) Confirm `probe` writes nothing to disk on any path.
- **Files:** `sgit_ai/sync/Vault__Sync.py` lines 1803–1837, `sgit_ai/cli/CLI__Vault.py`.
- **Evidence type:** Code read + stderr capture in tests.
- **Adversarial mutation:** Make `probe` write `clone_mode.json` — should be
  detected by a "no disk artefact" test.

### F05 — Delete-on-remote
- **Question:** Does the API truly destroy ciphertexts (or just the index)?
  Local key material wipe — confirmed *not* required (debrief says working copy
  preserved intentionally). Confirm: idempotent? Authenticated by write key?
  Does it require simple-token guard like `probe`? Does the confirmation prompt
  accept the vault_id from stdin without echoing?
- **Files:** `sgit_ai/sync/Vault__Sync.py` `delete_on_remote` (line 1724),
  `sgit_ai/api/Vault__API.py`, `sgit_ai/cli/CLI__Vault.py` (`--yes` flag).
- **Evidence type:** API contract reading + integration trace if cheap.
- **Adversarial mutation:** Strip write-key authentication on the API call —
  detected by an authorisation test?

### F06 — Sparse clone + on-demand fetch / cat
- **Question:** When a single file is `cat`-ed or `fetch`-ed, the server learns
  exactly which encrypted blob_id was accessed. Cross-correlated with deterministic
  tree IDs (F01), this lets a server map agent activity to subtrees. (a) Is this
  documented? (b) Do progress messages print *plaintext file paths* during fetch?
  (c) Does `cat --json` ever leak content_type / size from tree metadata that the
  user did not authorise?
- **Files:** `sgit_ai/sync/Vault__Sync.py` `fetch`, `cat`, `Vault__Storage.clone_mode_path`,
  `sgit_ai/cli/CLI__Vault.py`.
- **Evidence type:** Code read + manual stdout capture; document side-channel
  as accepted risk.

### F07 — `clone_mode.json` plaintext key persistence (read-only / share clones)
- **Question:** `Vault__Sync._clone_resolve_simple_token` writes a JSON file
  containing `read_key=read_key_hex` (line 1550, 1654). This is **plaintext key
  material on disk**, no passphrase wrap, no OS-keychain. For share-token clones
  this is intentional (the token itself derives the read_key) but for end users
  this is a clear forensic-disk leak.
- **Files:** `sgit_ai/sync/Vault__Sync.py` lines 1485, 1550, 1653–1656,
  `sgit_ai/sync/Vault__Storage.py` (`clone_mode_path`).
- **Evidence type:** Direct file read in a temp clone.
- **Adversarial mutation:** Replace `read_key_hex` with empty string — should
  break read-only clone tests, confirming the field is load-bearing.
- **Severity:** Medium-High; needs explicit doc that clone dirs are equivalent
  to plaintext-credential files.

### F08 — Surgical `write` encryption-path equivalence
- **Question:** `write_file()` builds a tree from a flat map without the
  full-scan path. Does it (a) call exactly the same `encrypt`/`encrypt_metadata_deterministic`
  primitives with the same `read_key`? (b) Skip any validation that `commit`
  performs (e.g., gitignore, large-blob threshold)? (c) Honour the read-only
  guard on share clones? (d) Update `content_hash_enc` consistently when content
  hash matches existing entry (no stale ciphertext bug)?
- **Files:** `sgit_ai/sync/Vault__Sync.py` `write_file`, `sgit_ai/sync/Vault__Sub_Tree.py`
  `build_from_flat`.
- **Adversarial mutation:** Replace `encrypt(read_key, content)` with
  `content` (no encryption) — must be detected by a test that opens
  `.sg_vault/objects/` and asserts ciphertext.

### F09 — Resumable push checkpoint contents
- **Question:** Debrief says `push_state.json` contains only object IDs.
  Verify by reading actual implementation. Does it persist any plaintext path,
  size, or commit message? Does it survive after Phase B success (must be
  cleared)? Does CAS conflict path leak anything?
- **Files:** `sgit_ai/sync/Vault__Sync.py` lines 963–1068, 2729–2742.
- **Evidence type:** File-content inspection on a real push.
- **Adversarial mutation:** Add `'paths': flat_map` to the checkpoint — detected
  by a "checkpoint contains only object IDs" test.

### F10 — Logging / error / progress hygiene
- **Question:** Did `CLI__Progress` (`scan`, `scan_done`, `commit`, `stats`
  phases) and `inspect_commit_chain` introduce plaintext file names, sizes,
  commit messages, or short blob_ids that, combined with deterministic tree
  IDs, are predictable from content? Specifically check the per-commit
  log line `"add report.pdf"` — that *is* a plaintext commit message, which
  is encrypted on the wire but printed locally; verify it never reaches a
  remote log sink (e.g., `--debug-log`, telemetry).
- **Files:** `sgit_ai/cli/CLI__Progress.py`, `sgit_ai/sync/Vault__Inspector.py`,
  `sgit_ai/cli/CLI__Debug_Log.py`.
- **Evidence type:** stdout/stderr capture + grep for plaintext fields in
  debug-log output.

### F11 — Token handling (probe, delete-on-remote, rekey)
- **Question:** Are tokens (passphrase:vault_id, simple tokens, share tokens)
  ever printed in full to logs, env vars, or process command lines? `rekey`
  prints the *new* vault_key on stdout (debrief shows `my-passphrase:a7f3c291e80b`)
  — confirm this is intentional and check for partial masking elsewhere. Verify
  no token is persisted to disk outside `vault_key` file.
- **Files:** `sgit_ai/cli/CLI__Vault.py`, `sgit_ai/cli/CLI__Token_Store.py`,
  `sgit_ai/transfer/Simple_Token.py`.

### F12 — Dependency CVE / version audit
- **Question:** Are `cryptography`, `osbot-utils`, and any transitive deps at
  versions free of known CVEs? Run `pip list --outdated` and cross-check against
  `cryptography` security advisories.
- **Files:** `pyproject.toml`, `requirements*.txt`.
- **Evidence type:** `pip-audit` or manual lookup; documented one-liner per dep.

## 4. Approach

1. **Read the seven sprint debriefs end-to-end** (done in this planning phase).
2. **Audit code statically** — start with the four files above, fan out to call
   sites only where ambiguity remains.
3. **Run existing unit tests** with `pytest -x tests/unit/` to confirm baseline
   green before any thinking about mutations.
4. **Design adversarial mutations on paper** (do not run); produce a mutation
   matrix linked to focus areas.
5. **Ask for handoff to QA/Architect** for issues that cross boundaries:
   - HMAC-IV crypto-architecture review → Architect
   - Type_Safe schema modelling for `clone_mode.json`, `push_state.json` → Dev
   - Mutation execution → QA (Phase 3)
6. **Write findings**, one Markdown file per focus area.

## 6. Adversarial Mutation Plan (for QA Phase 3)

In addition to the role-doc baseline (PBKDF2 iterations, AES key size, HKDF
info prefix, vault_key validation, skip-encryption-in-push):

| # | Mutation | Target test |
|---|----------|-------------|
| M1 | Replace `hmac.new(key, plaintext, sha256)` with `sha256(plaintext)` in `encrypt_deterministic` | Cross-vault IV reuse / IND-CPA test |
| M2 | Hard-code HMAC key to constant in `encrypt_deterministic` | Cross-vault tree-ID divergence test |
| M3 | Replace `iv=hmac(...)` with `iv=os.urandom(12)` in `encrypt_deterministic` | Determinism test (same flat → same tree_id) |
| M4 | In `rekey_wipe`, replace `shutil.rmtree` with no-op | "old keys gone after rekey" test |
| M5 | In `_pbkdf2_cached`, set `maxsize=0` (effectively disabling cache) | Performance test should still pass; correctness test must |
| M6 | In `clone_mode.json` write path, omit `read_key` field | Read-only clone read test must fail |
| M7 | In `write_file`, replace `encrypt(read_key, content)` with identity | "blob is encrypted" test must catch |
| M8 | In `push_state.json` save, add `paths` field | "checkpoint contents are object-id-only" test must catch |
| M9 | In `probe_token`, leak `vault_id` to debug log on success | "probe writes nothing" test must catch |
| M10 | In `delete_on_remote`, drop write-key auth header | API authorisation test must catch |

  do not attempt the bump from AppSec.
