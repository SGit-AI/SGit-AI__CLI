## 2. Priority reset

**Test infrastructure FIRST. Everything else SECOND.**

The reasoning, captured for posterity: we are about to add many more
scenarios (mutation matrix, crypto-determinism tests, push-checkpoint tests,
sparse-clone tests, plus future ones). If the test framework is slow or
fixture-bloated, every future agent + the CI pipeline pays the price on
every run. Speed and efficiency of the test suite is therefore a force
multiplier.

The test infrastructure work has two parts:

### Part (a) — Measure (Phase A)

Capture **before** any change is made:

- Current coverage (`pytest --cov` snapshot, per-module).
- Current full-suite runtime (cold, warm).
- Per-file runtime distribution (top-N slowest files).
- Per-test runtime outliers (top-N slowest individual tests).
- Duplicate setup work across files (vault creation, key derivation, temp
  dirs, etc.).

These numbers are the baseline. Every change after this point gets
measured against them.

### Part (b) — Improve (Phase B)

Design and implement:

- **Shared / pre-built fixtures** — real objects (no mocks), session-scoped
  where safe, that eliminate repeated expensive setup. **Critical
  constraint:** pre-built state must NOT compromise what's being tested.
  No shared mutable state hiding bugs. No fixture that masks a real
  failure mode. If a test relies on "fresh state for every run", it stays
  fresh — only setup that is genuinely behaviour-irrelevant gets shared.
- **Test parallelization** where safe (pytest-xdist), with explicit
  documentation of which tests are not parallel-safe and why.
- **Reduce per-test setup** for tests where setup dominates assertion
  cost.

### Acceptance criteria — Phase B exit

Concrete numeric targets are set in `06__test-infra-acceptance.md`. Until
these are met, **no other brief in this pack starts.**

Each brief is small (≤200 lines target), self-contained, and explicitly
states:
- **Owner role** (Dev / DevOps / Architect / AppSec / QA, primary).
- **Prerequisites** — which other briefs must be merged first.
- **Scope** — exactly what files to touch.
- **Out of scope** — files / behaviour the agent must not change.
- **Acceptance criteria** — pass/fail conditions the agent verifies before declaring done.
- **Deliverables** — code, tests, docs, commits.
- **Estimated effort** — rough time for the executor agent.

## 5. What this plan deliberately does NOT do

- **Does not assign agents.** Each brief is picked up by a Sonnet session
  when Dinis decides to launch it.
- **Does not change source or tests.** This entire planning effort is
  read-only and document-only.
- **Does not commit to specific numeric targets in this file.** Numbers
  belong in the briefs (`01__`, `02__`, `06__`) where the executor agent
  reads them.
- **Does not pre-decide HOW the deferred queue is sequenced after Phase B
  exit.** That decision happens after we see Phase B's actual results.
