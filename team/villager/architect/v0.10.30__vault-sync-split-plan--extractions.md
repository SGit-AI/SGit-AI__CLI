## E1 — `_build_dir_walk_core` (internal Sub_Tree refactor)

**Target file:** `sgit_ai/sync/Vault__Sub_Tree.py` (same file, no new module)
**Risk:** LOW — no public surface changes.

**Problem:** `Vault__Sub_Tree.build` (lines 24–113) and `build_from_flat`
(lines 115–188) share ~80 lines of identical dir-walk logic (the
`dir_contents`/`all_dirs` population + `sorted_dirs` tree-assembly loop).
The only difference is how each leaf entry is produced.

**Proposed method:**
```python
def _build_tree_from_dir_contents(
    self,
    dir_contents: dict,
    all_dirs: set,
    make_entry: callable,   # (filename, rel_path) -> Schema__Object_Tree_Entry | None
    read_key: bytes,
) -> str:
    """Shared tree-assembly core. Returns root tree object ID."""
```

`build` and `build_from_flat` each become ~25-line wrappers that populate
`dir_contents`/`all_dirs` then call `_build_tree_from_dir_contents`.

**Test strategy:** no new tests required; existing `Vault__Sub_Tree` tests
cover both `build` and `build_from_flat`. If coverage drops, add direct tests.

**Callers changed:** none externally; `build` and `build_from_flat` call the
new private method.

## E3 — `Vault__Graph_Walk` (BFS unification)

**Target file:** `sgit_ai/sync/Vault__Graph_Walk.py` (NEW)
**Risk:** MEDIUM — two callers with different semantics; must preserve
`stop_at` and `include_blobs` parameters.

**Problem:** BFS commit/tree/blob walk appears in:
1. `_clone_with_keys` lines 1356–1408 (clone path; no `stop_at`)
2. `_fetch_missing_objects` lines 2384–2459 (pull path; `stop_at` set,
   `include_blobs` flag)

**Proposed class:**
```python
class Vault__Graph_Walk(Type_Safe):
    obj_store   : Vault__Object_Store
    read_key    : bytes               = None
    stop_at     : set                 # commit IDs where BFS stops
    include_blobs : bool              = True
    on_commit   : callable            = None  # (commit_id, commit_obj) -> None
    on_tree     : callable            = None  # (tree_id) -> None
    on_blob     : callable            = None  # (blob_id, is_large) -> None
    on_progress : callable            = None  # (phase, msg, detail) -> None

    def walk(self, start_commit_ids: list) -> dict:
        """BFS from start_commit_ids. Returns {n_commits, n_trees, n_blobs}."""
```

`_clone_with_keys` configures `Vault__Graph_Walk` with `stop_at=set()`,
`include_blobs=True`.
`_fetch_missing_objects` configures with `stop_at=<locally-present commits>`,
`include_blobs=<flag>`.

**Visited-set unification:** the clone path uses `visited_commits +
visited_trees`; the pull path uses `visited_commits + seen_trees + seen_blobs`.
`Vault__Graph_Walk` owns all three internally.

**Commit-message decryption:** `_fetch_missing_objects` decrypts the commit
message for the per-commit log line (lines 2484–2492). This is a pull-only
concern. Pass an `on_commit` callback; the pull caller supplies it.

**Test strategy:** add `tests/unit/sync/test_Vault__Graph_Walk.py` with:
- `test_walk_single_commit_single_blob` — minimal vault walk
- `test_walk_stop_at_respects_boundary` — pull stop_at semantics
- `test_walk_sparse_skips_blobs` — `include_blobs=False`
- `test_walk_callbacks_invoked` — on_commit / on_tree / on_blob fired

**Problem:** Small/large blob bucketing + concurrent download appears in:

Same bucketing logic (`CLONE_LAMBDA_SAFE_BYTES`, `MAX_RESPONSE_BYTES`,

**Proposed method on `Vault__Object_Store`:**
```python
def batch_download(
    self,
    vault_id: str,
    blob_file_ids: list,       # list of 'bare/data/{blob_id}' strings
    is_large_flags: list,      # parallel list: True if large
```

## E5 — Class-level split of `Vault__Sync.py`

**Target:** split into 8 new sub-class files + thin `Vault__Sync` delegation facade.
**Risk:** HIGH. Start only after E1–E4 green **and Dinis sign-off.**

### Delegation pattern (replaces MRO approach)

Each sub-class is an independent `Type_Safe` class that receives `crypto`
and `api` as constructor arguments. `Vault__Sync` creates instances on
demand and delegates calls. No multiple inheritance; no MRO risk.

```python
# sgit_ai/sync/Vault__Sync__Commit.py
class Vault__Sync__Commit(Type_Safe):
    crypto : Vault__Crypto = None
    api    : Vault__API    = None

    def commit(self, directory: str, message: str = '') -> dict: ...
    def write_file(self, directory: str, path: str, content: bytes, ...) -> dict: ...

# sgit_ai/sync/Vault__Sync.py (facade — ~150 LOC after split)
class Vault__Sync(Type_Safe):
    crypto : Vault__Crypto
    api    : Vault__API

    def commit(self, directory: str, message: str = '') -> dict:
        return Vault__Sync__Commit(crypto=self.crypto, api=self.api).commit(directory, message)

    def write_file(self, directory: str, path: str, content: bytes, ...) -> dict:
        return Vault__Sync__Commit(crypto=self.crypto, api=self.api).write_file(...)
```

A new sub-class instance is created per call — negligible overhead for a CLI.
Sub-classes are independently testable: `Vault__Sync__Commit(crypto=c, api=a)`.

### State carrier: `Vault__Components`

`Vault__Components` (already exists) carries all per-vault state: `vault_id`,
`read_key`, `write_key`, `sg_dir`, `storage`, `obj_store`, `ref_manager`,
`branch_manager`, etc. Sub-classes call `self._init_components(directory)` via
a helper — `_init_components` lives in `Vault__Sync__Init` and is passed to
sub-classes that need it, or each sub-class re-implements its own
`_init_components` call using the same logic.

**Preferred approach:** keep `_init_components` on a shared mixin base that
all sub-classes can call:

```python
class Vault__Sync__Base(Type_Safe):
    crypto : Vault__Crypto = None
    api    : Vault__API    = None

    def _init_components(self, directory: str) -> Vault__Components:
        ...  # moved here from Vault__Sync

class Vault__Sync__Commit(Vault__Sync__Base): ...
class Vault__Sync__Pull(Vault__Sync__Base):   ...
```

`Vault__Sync__Base` is the ONLY Type_Safe base across the hierarchy — no
multiple-`Type_Safe` inheritance anywhere.
